# `F_CMD_FSIO_CHECK` 确认时间问题调试复盘

## 1. 问题起点 / 需求背景

### 最初的问题或需求是什么

这次需求的起点很明确：

- 现有注塑检测动作里已经有一个“延时”
- 现在还要再加一个“确认时间”
- 这个“确认时间”不是单纯展示字段，而是要参与检测滤波

用户已经给出了几条关键线索：

- 上位机相关入口在 `qml/App_HAMOUI/teach/extents/ExtentActionDefine.js`
- 下位机处理在 `F_CMD_FSIO_CHECK`
- 上下位机之间的连接点就是这条指令

也就是说，这不是单纯改一个界面输入框，而是一条完整链路问题：UI 输入 -> HMI 动作定义 -> 协议下发 -> Host 解析 -> 运行时检测。

### 初始现象 / 期望实现的功能效果

当时的核心现象是：

- UI 上其实已经能看到“确认时间”输入框
- 但实际动作链路里没有这个字段
- 下位机检测逻辑里也没有对应滤波处理

期望的效果是：

- 开始检测到结束检测这一段时间内持续监控
- 如果信号中途短暂异常，但异常持续时间没有超过确认时间，就不报警
- 只有异常连续保持超过确认时间，才触发报警

### 当时最可能的几个原因或实现思路

一开始最可能的思路有 3 个：

1. UI 已经有“确认时间”输入框，但只是半成品，没有真正接到动作对象。
2. `F_CMD_FSIO_CHECK` 目前只有 `delay`，需要扩协议，把 `confirmTime` 一起下发。
3. 也可能不用改协议，而是在下位机写死一个滤波时间，或者复用原有字段。

当时还不能直接拍板，因为不知道“延时”和“确认时间”分别落在哪一层，也不知道下位机现在到底是“单次判断”还是“后台持续检测”。

---

## 2. 分析与排查过程（按时间顺序）

### 步骤 1

#### 当时的判断是什么

先不要猜实现方式，先把整条链路摸清楚：`F_CMD_FSIO_CHECK` 到底是什么动作，`delay` 现在在哪一层生效，真正报警是在哪里做的。

#### 为什么这么判断（依据）

这种需求最容易犯的错，就是只盯着 UI 或只盯着下位机，结果改了一半。

如果不知道这条指令在运行时真实做什么，就很容易：

- 把确认时间加错层
- 误以为 `delay` 就是滤波
- 或者把“开始检测延时”和“异常确认时间”混为一谈

#### 做了什么验证

我先读了这几类文件：

- 需求文档和现有分析文档
- HMI 的 `ExtentActionDefine.js`
- 快编面板 `EasyProgramFunctionPanel.qml`
- 注塑工艺生成页 `InjectTechnology.qml`
- Host 的 `injecttechnologytask.h/.cpp`

#### 结果如何

第一轮确认出了两个关键事实：

1. `delay` 不是滤波时间，而是“延时多久后把检测挂上去”。
2. 真正持续检测并触发报警的，不在 `CheckFSAction::process()`，而在 `InjectTechnologyTask::FSSignalCheck()`。

这一步很关键，因为它把问题从“动作触发时怎么判断”转成了“后台持续检测怎么滤波”。

---

### 步骤 2

#### 当时的判断是什么

既然持续检测在 `FSSignalCheck()`，那“确认时间”大概率也应该落在这里，而不是另起一条等待动作。

#### 为什么这么判断（依据）

`CheckFSAction::process()` 的逻辑非常简单：

- 记录开始时间
- 到了 `delay` 就调用 `SetFSSignalCheckInit()`
- 然后动作结束

它本身不持续读 IO，也不持续报警。

所以如果要实现“检测区间内异常持续多长时间才报警”，最自然的落点一定是后台轮询函数 `FSSignalCheck()`。

#### 做了什么验证

我继续看了：

- `SetFSSignalCheckInit()`
- `FSSignalCheck()`
- `AutoActionStart()`
- `Init()`

#### 结果如何

确认当前下位机逻辑是：

- `isCheck = 1` 时，表示这一路进入检测状态
- `FSSignalCheck()` 每个周期直接读当前 IO
- 如果当前值不符合 `xdir` 预期，就立刻报警

也就是说，原逻辑完全没有“异常持续时间”这个概念，只有“当前周期是否异常”。

---

### 步骤 3

#### 当时的判断是什么

既然运行时需要 `confirmTime`，那 HMI 也必须真的把这个字段带下来，不能只留在界面里。

#### 为什么这么判断（依据）

如果 HMI 不下发 `confirmTime`，那下位机根本不知道每条动作的滤波时长是多少。

而且用户已经指出 UI 里“目前是已经有了个延时的，现在需要加上这个确认时间做滤波用”，这说明操作入口最好还是沿用现有 UI。

#### 做了什么验证

我检查了：

- `EasyProgramFunctionPanel.qml` 里“确认时间”控件是否有 `id`
- 是否会回填 `actionObject.confirmTime`
- 是否会参与 `generateFSCheckAction.generateActionObject(...)`
- `ExtentActionDefine.js` 的 `generateFSCheckAction` 是否定义了 `confirmTime`

#### 结果如何

发现了一个很典型的“界面半成品”现象：

- 界面上已经摆了“确认时间”输入框
- 但它没有 `id`
- 不会回填
- 不会生成到动作对象
- `generateFSCheckAction` 也没有这个字段

结论很明确：上位机这部分只是 UI 看起来像有功能，实际链路还没接通。

---

### 步骤 4

#### 当时的判断是什么

协议要不要兼容旧动作数据，是一个必须先决定的分叉点。

#### 为什么这么判断（依据）

我看到 HMI 自定义动作编译是按 `properties` 顺序下发的，而 Host 解析是严格按 `GetParaLen()` 和固定顺序取字段。

这意味着只要改字段顺序或长度，就会直接影响旧动作数据能不能继续解析。

#### 做了什么验证

我检查了：

- `CustomActionCompiler()` 的行为
- `Teach.js` 里 `toRegisterString()` 的行为
- Host 侧对动作长度的严格校验

#### 结果如何

确认了两件事：

1. 自定义动作确实是按字段顺序顺次编译的。
2. Host 解析长度是严格校验的，旧长度和新长度不匹配会直接报教导数据错误。

所以这里没有“随便加个字段也许能兼容”的空间，必须明确是兼容方案还是非兼容方案。

---

### 步骤 5

#### 当时的判断是什么

在讨论后，先按“直接扩字段，不兼容旧二进制动作”的方案实现。

#### 为什么这么判断（依据）

这个方案最直接，思路也最清晰：

- `F_CMD_FSIO_CHECK` 增加一个独立 `confirmTime`
- 上位机直接下发
- 下位机直接解析

从工程实现角度看，这通常是最不容易出隐性 bug 的做法。

#### 做了什么验证

我按这个思路先改了：

- HMI 动作定义
- 快编面板
- 注塑页直接生成动作的地方
- Host 的解析和运行时逻辑

#### 结果如何

功能上是能成立的，但我在 Host 侧为了让结构更清楚，第一次把 `fsSignalFlag` 改成了“配置 + 运行态”结构体，这带来了一个新的问题：

- 代码改动面太大
- 用户一眼就看出和原代码风格差异太大
- 对现场代码维护来说，不够稳

这一步功能方向没错，但实现方式偏重了。

---

### 步骤 6

#### 当时的判断是什么

下位机必须回到“最小改动”思路，尽量保留原逻辑骨架，只在必要位置插入滤波。

#### 为什么这么判断（依据）

用户明确指出：

- 下位机不要做太大改动
- 只需要增加确认时间做滤波

这其实不是单纯代码风格偏好，而是现场项目里很现实的要求：

- 改动越大，回归成本越高
- 越接近原逻辑，越容易对照验证
- 出问题时也更容易回滚

#### 做了什么验证

我回看了自己第一次的 Host diff，确认哪些改动是“必须的”，哪些只是为了“写得更整洁”。

#### 结果如何

最终把 Host 收缩成了更小的版本：

- `fsSignalFlag` 仍然保留原来的 `FSCHECK fsSignalFlag[FS_SINGLE_MAX]`
- 只额外新增两个并行数组：
  - `fsSignalConfirmTime[]`
  - `fsSignalAbnormalSince[]`
- `FSSignalCheck()` 仍然沿用原来的 `if (isCheck) -> if (xdir == 1 / 0)` 骨架
- 只是在“发现异常”的地方加了“先计时，再超时报警”

这一步把“功能正确”和“改动最小”两件事同时兼顾了。

---

### 步骤 7

#### 当时的判断是什么

虽然代码实现上已经是“异常持续超过确认时间才报警”，但还要再确认一次语义，避免我和用户对“确认时间”的理解不一致。

#### 为什么这么判断（依据）

这类需求里最容易出错的不是代码，而是词义：

- 有的人说“确认时间”，其实指“开始前延时”
- 有的人说“确认时间”，其实指“异常滤波”
- 有的人说“确认时间”，其实指“正常信号要稳定多久才算成立”

如果词义没对齐，代码即使跑了，也会被认为“用反了”。

#### 做了什么验证

用户在测试后明确描述了真实业务语义：

- 在开始检测到结束检测的整段时间里持续监控
- 中间可能掉信号
- 但只要掉信号时间没有超过确认时间，就不报警

#### 结果如何

对照代码后确认：

- 现在实现的逻辑，实际上正符合这个说法
- 它滤的是“异常持续时间”
- 不是“开始检测前延时”
- 也不是“正常信号确认时间”

这一步的价值不在于改代码，而在于把“代码做了什么”和“业务希望它代表什么”彻底对齐。

---

## 3. 过程中出现的新问题

### 新问题 1：第一次下位机改动过大

#### 出现原因

我第一次实现时，把 `fsSignalFlag` 从原来的简单数组改成了“配置 + 运行态”的结构体。

这样写从抽象上更整齐，但对这个项目来说，改动面超出了“最小修复”的合理范围。

#### 这个问题本质属于

- 逻辑问题：不是逻辑错，而是实现策略不合适
- 其他：属于“工程改动尺度控制”问题

### 新问题 2：一度担心 `confirmTime` 字段插入顺序会错位

#### 出现原因

`F_CMD_FSIO_CHECK` 原本只有固定字段顺序，现在新增字段后，如果上下位机顺序不一致，就会整体错位。

这个问题不是代码写错，而是协议设计天然敏感。

#### 这个问题本质属于

- 通信问题

### 新问题 3：对“确认时间”语义的误读风险

#### 出现原因

“确认时间”这个名字很容易让人想到“开始检测前确认一下”，但现场真实语义是“异常滤波时间”。

如果只看名字不看用户描述，很容易在逻辑上加反位置。

#### 这个问题本质属于

- 逻辑问题
- 其他：更准确地说，是需求语义理解问题

## 4. 关键转折点（最重要）

### 哪一步让问题真正接近解决

真正的关键转折点，是确认：

- `F_CMD_FSIO_CHECK` 不是一次性立即判断
- 真正持续检测的是 `InjectTechnologyTask::FSSignalCheck()`

### 为什么这一点是关键

因为只要这一步认错了，后面所有实现都会歪：

- 会去 `CheckFSAction::process()` 里做复杂判断
- 会想加一条额外等待动作
- 会把确认时间误做成“开始前延时”

而一旦明确后台持续检测点就是 `FSSignalCheck()`，整个问题就变简单了：

- `delay` 负责“何时开始挂监控”
- `confirmTime` 负责“异常持续多久才报警”

两者职责就分清楚了。

### 如果没有这一步，会一直卡在哪

如果没有这一步，后面会一直卡在两个误区里：

1. 反复纠结 `delay` 和 `confirmTime` 到底谁是“开始前等待”
2. 试图在动作执行阶段就做完所有判断，而不是在后台监控阶段做滤波

---

## 5. 最终解决方案

### 最终是怎么修复的（核心思路）

最终方案其实很简单：

- 上位机把 `confirmTime` 真正编进 `F_CMD_FSIO_CHECK`
- 下位机解析这个字段
- 后台检测时，只对“异常持续时间”做滤波

### 涉及的关键点（3 点以内）

1. `delay` 和 `confirmTime` 分工明确  
   `delay` 负责延时挂检测，`confirmTime` 负责异常滤波。

2. 下位机最小改动实现  
   不重构原检测结构，只增加并行数组存确认时间和异常起始时间。

3. 语义重新对齐  
   明确“确认时间”不是开始前等待，而是异常持续多久才真正报警。

---

## 6. 如果重新来一遍，最快路径是什么

### 可以跳过哪些无效尝试

如果重来一次，最该跳过的是：

- 一开始就做较大规模的 Host 结构重构
- 过早讨论“代码怎么写得更漂亮”

这类现场项目里，先满足“改动小、逻辑准、能回归”更重要。

### 最优排查顺序应该是怎样

最快路径应该是：

1. 先确认 `F_CMD_FSIO_CHECK` 的真实职责  
   是单次判断，还是后台持续检测。

2. 再确认 `delay` 的职责  
   是开始前延时，还是滤波时间。

3. 然后确认 UI 上“确认时间”有没有真正编进动作对象。

4. 最后只在 `FSSignalCheck()` 原逻辑上补最小滤波判断。

这样基本不会走弯路。

---

## 7. 本质总结（非常重要）

### 这个问题的“本质原因”一句话总结

这个问题的本质，不是“少了一个字段”，而是“`confirmTime` 的业务语义没有在整条动作链路里被一致实现”。

### 以后遇到类似问题，可以优先检查什么

以后遇到类似问题，优先检查这几件事：

1. 这个参数到底属于“动作触发时”还是“后台持续运行时”。
2. UI 上有输入框，不代表它真的参与了动作编译。
3. 看到“确认时间 / 延时 / 滤波时间”这类词，先问清它滤的是“开始时机”还是“异常持续时间”。
4. 现场项目里，先选最小改动方案，再考虑重构。

如果把这四点先想清楚，类似问题会快很多。
