# 本次问题复盘

## 1. 问题起点 / 需求背景

最初的需求是：`InjectSpeedControlArea.qml` 里几个固定预设按钮，不仅要能点击直接使用固定值，还要支持长按后自定义修改数值。

期望效果有两层：

* 长按按钮后，可以输入新的预设值
* 修改完成后，按钮显示要跟着变化；但真正生效要等用户再点一次按钮，而不是点“确定”就立刻生效

在最开始，最可能的几个方向是：

* 方向 1：项目里已经有“长按”信号和输入弹窗，优先复用现有机制，不新造控件
* 方向 2：修改的是“按钮预设值”还是“当前生效值”，这两个语义要先分清
* 方向 3：按钮文字原来靠前导空格做对位，这种写法可能在数值动态变化后出现显示问题

## 2. 分析与排查过程（按时间顺序）

### 步骤 1：

* 当时的判断是什么  
  先确认这几个按钮现在怎么工作的，以及项目里有没有现成的“长按”能力。
* 为什么这么判断（依据）  
  这种需求最怕一上来就自己造一套交互，结果项目里其实已经有统一写法。先摸清现状，能减少无效改动。
* 做了什么验证（看代码 / 打日志 / 改代码）  
  看了 `InjectSpeedControlArea.qml` 对应按钮区域，也全局搜索了 `onPressAndHold`、`btnPressAndHold`、`ICButton`。
* 结果如何  
  确认 `ICButton` 本身就支持 `btnPressAndHold()`，并且项目里已有 `ICMessageBox.showInput(...)` / `virtualKeyboard` 这种现成输入方式，可以直接复用。

### 步骤 2：

* 当时的判断是什么  
  既然按钮支持长按，最自然的实现是：短按继续走原逻辑，长按弹输入框，修改这个按钮对应的预设值。
* 为什么这么判断（依据）  
  这样改动范围最小，不需要改外部交互模型，也不用新增新组件。
* 做了什么验证（看代码 / 打日志 / 改代码）  
  先写了一版：给 5 个按钮加长按输入，并把新值保存到本地配置里，保证页面重新打开后也能记住。
* 结果如何  
  功能是能跑的，但这是“功能完整版”，不是“最小逻辑版”。

### 步骤 3：

* 当时的判断是什么  
  用户明确说要“最小逻辑版”，而且提到原来的按钮文字是靠空格做对位，所以前一版做得偏重了。
* 为什么这么判断（依据）  
  用户的真实优先级不是“永久保存”，而是“先把长按改值做出来，而且尽量少动现有结构”。
* 做了什么验证（看代码 / 打日志 / 改代码）  
  把持久化读写整套逻辑删掉，只保留页面内的临时预设值；同时保留原先“靠空格调位”的文字思路。
* 结果如何  
  这一步把实现收缩到了更接近需求的版本，但后面暴露出两个问题：输入值异常、按钮文本异常。

### 步骤 4：

* 当时的判断是什么  
  出现了“输入 10 变成 1，输入 50 变成 5”这种现象，优先怀疑不是按钮逻辑错，而是输入控件的小数处理方式有坑。
* 为什么这么判断（依据）  
  这个症状很像某种“把整数按一位小数解释”的行为。例如输入 `10`，内部被当成 `1.0`。
* 做了什么验证（看代码 / 打日志 / 改代码）  
  追到了 `ICMessageBox -> ICConfigEdit -> ICLineEdit` 这条链，发现数值输入会带 `decimal` 规则。于是把“速度预设”的输入从 `decimal: 1` 改成 `decimal: 0`。
* 结果如何  
  这个问题被修掉了。输入 `10`、`20`、`50` 终于不会再被缩成 `1`、`2`、`5`。

### 步骤 5：

* 当时的判断是什么  
  长按修改后，代码里把新值立刻应用到了当前速度 / 当前寸动距离，但用户明确说这不对。
* 为什么这么判断（依据）  
  用户想要的是“改按钮预设”，不是“立即执行按钮点击”。
* 做了什么验证（看代码 / 打日志 / 改代码）  
  去掉了长按确认后立刻调用 `setRealSpeed(...)` 和 `applyInchDistance(...)` 的逻辑。
* 结果如何  
  交互行为变正确了：长按只改按钮值，真正生效要再点一次按钮。

### 步骤 6：

* 当时的判断是什么  
  出现“实际点击已经按新值生效，但按钮文字还是旧值”的现象，第一反应是 QML 对数组元素的绑定刷新不稳定。
* 为什么这么判断（依据）  
  点击逻辑读的是数组里的新值，说明数据已经更新；但文字没变，说明可能是绑定没重算，或者 UI 没刷新。
* 做了什么验证（看代码 / 打日志 / 改代码）  
  当时先做了一个保守修正：给按钮加 `id`，新增显式 `refreshQuickPresetTexts()`，手动给按钮 `text` 赋值。
* 结果如何  
  这一步看起来像是在修“刷新问题”，但后来证明它没有打到根上。因为文本不是“没刷新”，而是“刷新出来的字符串本身就错了”。

### 步骤 7：

* 当时的判断是什么  
  用户反馈一个很关键的现象：`25`、`31`、`62` 这些不以 `0` 结尾的值显示正常，而 `20`、`30` 这种以 `0` 结尾的值显示成了 `2`、`3`。
* 为什么这么判断（依据）  
  这个现象非常强，说明问题不是布局宽度，也不是点击逻辑，而是字符串格式化阶段把末尾 `0` 错误删掉了。
* 做了什么验证（看代码 / 打日志 / 改代码）  
  在“长按确认后”和“真正显示的 `Text` 控件上”分别加了日志，打印：
  * 输入值
  * 存进去的数值
  * 生成出来的按钮文本
  * 显示层 `Text.text`
* 结果如何  
  日志直接证明：`input=20`，`value=20`，但 `generated=2%`。这说明问题根本不在渲染层，而在 `formatPresetValue()`。

### 步骤 8：

* 当时的判断是什么  
  `formatPresetValue()` 原本想做“去掉无意义尾零”，但实现有缺陷：`20.0 -> 20. -> 20 -> 2`。
* 为什么这么判断（依据）  
  日志已经把“值正确、生成字符串错误”这个事实钉死了，剩下只要看格式化函数本身。
* 做了什么验证（看代码 / 打日志 / 改代码）  
  先修了一版循环逻辑，只删小数部分的尾零，不再误删整数位的 `0`。
* 结果如何  
  逻辑上已经能修掉 `20 -> 2` 的问题，但这时又考虑到：这个“去尾零”本身不是刚需，而且它已经引入过一次问题。

### 步骤 9：

* 当时的判断是什么  
  既然用户明确说“保留一位小数 0 也没问题”，那最稳的方式就是彻底不要做“去尾零”。
* 为什么这么判断（依据）  
  一个功能如果不是必须存在，却明显增加了错误概率，就应该删掉，而不是继续补丁修补。
* 做了什么验证（看代码 / 打日志 / 改代码）  
  把 `formatPresetValue()` 简化成直接 `Number(value).toFixed(decimal)`。
* 结果如何  
  文本生成逻辑彻底变简单，像 `20.0%`、`30.0%` 这种显示虽然比以前多一个 `.0`，但稳定、可预测、不容易再出错。

### 步骤 10：

* 当时的判断是什么  
  另一个独立问题是：按钮原来靠前导空格做视觉对位，这种写法和 `ICButton` 里的 `Text` 布局并不匹配。
* 为什么这么判断（依据）  
  `ICButton` 内部是按文本真实宽度居中，而且字体是比例字体，不是等宽字体，所以靠空格“假装对齐”天然不稳。
* 做了什么验证（看代码 / 打日志 / 改代码）  
  不再让按钮的 `text` 直接承载显示内容，而是：
  * 按钮本身 `text: ""`
  * 在按钮内部单独放一个 `Text`
  * 用 `horizontalAlignment: Text.AlignHCenter` 和 `verticalAlignment: Text.AlignVCenter` 做真实居中
* 结果如何  
  这个改法把“文本内容”和“布局方式”分开了。以后不管是 `20.0%`、`25.0%` 还是 `100.0%`，都由布局系统做居中，而不是靠空格硬凑。

## 3. 过程中出现的新问题

### 新问题 1：一开始做成了“持久化版本”

* 出现原因  
  我一开始默认“用户修改预设值”通常意味着要保存下来，所以顺手加了配置读写。
* 本质类型  
  逻辑问题 / 需求理解偏重

### 新问题 2：长按确认后立即生效

* 出现原因  
  我把“修改预设值”和“应用当前值”混在了一起，默认长按确认后应该立刻看到效果。
* 本质类型  
  逻辑问题 / 状态问题

### 新问题 3：误以为是“UI 没刷新”，实际是“生成文本就错了”

* 出现原因  
  当看到“点击已经按新值生效，但文字没对”时，第一反应容易放到绑定刷新上；这属于一个合理但不够深入的猜测。
* 本质类型  
  UI问题

### 新问题 4：为了排查，临时尝试过“改按钮宽度”

* 出现原因  
  这是一个典型的基于经验的快速猜测：怀疑文本被裁掉或遮挡。但用户给出“25 正常、20 异常”的信息后，这条路基本就被排除了。
* 本质类型  
  UI问题

### 新问题 5：`formatPresetValue()` 的“去尾零”逻辑反而制造了新 bug

* 出现原因  
  这是最关键的教训。这个函数原意是让显示更“好看”，但实现上没有把“删小数部分尾零”和“保留整数位尾零”区分开。
* 本质类型  
  逻辑问题

## 4. 关键转折点（最重要）

真正让问题接近解决的关键一步，是加日志后发现：

* `input=20`
* `value=20`
* 但 `generated=2%`

为什么这一点最关键：

* 它把问题从“可能是 UI 刷新问题 / 可能是控件布局问题 / 可能是数据没更新”一下缩小到了“字符串生成阶段就已经错了”
* 这一步直接砍掉了很多模糊猜测，让后面的排查变成“只看格式化函数”

如果没有这一步，就很容易继续在这些方向里打转：

* 怀疑 QML 绑定不更新
* 怀疑按钮宽度不够
* 怀疑 `Text` 渲染裁剪

## 5. 最终解决方案

最终的修复核心不是“加更多补丁”，而是把问题拆开，一层一层简化：

* 输入层：速度预设输入改成整数模式，避免 `10 -> 1` 这类错误
* 状态层：长按只改预设，不立即生效；真正生效仍然靠点击按钮
* 显示层：按钮文本不再靠前导空格排版；并且去掉了危险的“去尾零”处理，直接按固定小数位显示

涉及的关键点只有 3 个：

* 区分“修改预设值”和“立即执行”
* 区分“数据问题”和“显示问题”
* 删除不必要、但会制造 bug 的“美化逻辑”

## 6. 如果重新来一遍，最快路径是什么

可以跳过的无效尝试：

* 不必一开始就做持久化
* 不必先尝试改按钮宽度
* 不必过早假设是 QML 绑定刷新问题

更优的排查顺序应该是：

1. 先确认长按输入链路是否正确拿到原始值  
2. 立刻打印“输入值 -> 存储值 -> 生成字符串 -> 显示文本”这四层  
3. 如果数据对、字符串错，直接查格式化函数  
4. 如果字符串对、显示错，再去看 `ICButton` 和 `Text` 布局

换句话说，最快路径不是“多试几种改法”，而是尽早把链路切成几段，分别验证。

## 7. 本质总结（非常重要）

这个问题的本质原因一句话总结：

* 一个本来只想“优化显示效果”的格式化逻辑，错误地修改了真实显示内容；同时旧的按钮排版方式又放大了问题，让排查一开始看起来像 UI 问题。

以后遇到类似问题，优先检查这几件事：

* 输入值有没有在进入业务逻辑前就被转换错
* “存储值”和“显示字符串”是不是同一个值
* UI 异常到底是“数据错了”还是“渲染错了”
* 凡是“美化显示”的逻辑，都要警惕它是否在偷偷改业务含义
