```C++
void IOStatusRowStopCheck() {
  IOSTATUS* tmp;
  for (int i = 0; i < IO_Status_Row.size(); i++) {
    tmp = (IOSTATUS*)IO_Status_Row.get(i);
    if (!(IsInEdgeRow(tmp->id, tmp->board) && tmp->type == 1)) {
      tmp->check = 0;
      tmp->rising_edge = 0;
      tmp->falling_edge = 0;
    }
  }
}
```

## 提示词学习
### 上下文长度达限，总结内容提示词

要求输出为 Markdown 文件内容，直接写入项目根目录的新建 .md 文件中，包含：
1. 问题背景
  - 当前要解决的bug/需求是什么
  - 预期行为 vs 实际问题
2. 当前进展
  - 已经做了哪些尝试
  - 改过哪些地方（简要列出文件和修改点）
3. 关键代码上下文
  - 与问题最相关的代码片段（精简但完整可理解，代码总长度控制在200行以内，只保留关键路径）
  - 标明文件路径
4. 问题卡点
  - 当前具体卡在哪里
  - 哪些现象还没解释清楚
5. 已排除的方向
  - 哪些思路已经试过但不行
6. 下一步建议
  - 如果你继续分析，会优先从哪里入手

要求：
- 内容尽量精简，但保证新窗口能“无缝接手”
- 不要长篇解释
- 代码只保留关键部分

### 调试过程复盘
请基于我们刚才整个问题解决过程，整理一份“调试过程复盘文档”，用于我学习你是如何一步步解决问题的。

写入项目根目录的新建 .md 文件，重点不是代码，而是“思考过程”，请参考下面的结构：
1. 问题起点 / 需求背景
	- 最初的问题或需求是什么
	- 初始现象（错误、日志、表现）或期望实现的功能效果
	- 当时最可能的几个原因或实现思路（列2~3个）
2. 分析与排查过程（按时间顺序）
	请按步骤列出你是如何一步步分析的：
	- 步骤 1：
		- 当时的判断是什么
		- 为什么这么判断（依据）
		- 做了什么验证（看代码 / 打日志 / 改代码）
		- 结果如何
	- 步骤 2：
		（保持同样结构）
		（一直写到最终解决）
3. 过程中出现的新问题
	- 在修改过程中引入了哪些新问题
	- 每个新问题出现的原因是什么（非常重要）
	- 这些问题本质属于：
	    - 逻辑问题 / 状态问题 / 线程问题 / 通信问题 / UI问题 / 其他
4. 关键转折点（最重要）
	- 哪一步让问题真正接近解决

	- 如果没有这一步，会一直卡在哪
5. 最终解决方案
	- 最终是怎么修复的（核心思路，不要堆代码）
	- 涉及哪几个关键点（列3点以内）
6. 如果重新来一遍，最快路径是什么
	- 可以跳过哪些无效尝试
	- 最优排查顺序应该是怎样
7. 本质总结（非常重要）
