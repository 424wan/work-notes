# EUY 输出执行链路与 I/O 监视页不刷新问题分析

## 1. 背景

这次排查的起点是两个连续的问题：

1. `qml/App_HAMOUI/teach/OutputActionEditor.qml` 里的 `EUY` 选项，插入教导指令以后，指令到底是怎么继续执行下去的。
2. 实际现场现象是：`EUY` 指令执行后，注塑机侧看起来已经有输出动作，但 HMI 顶部 `I/O` 页面里的 `EU输出` 却没有同步显示出来。

为了把问题查清楚，排查分成了两条线：

- 第一条线：先确认 `EUY` 指令是不是**真的执行到了主机并驱动了 EU 输出**。
- 第二条线：再确认 `I/O` 监视页显示的状态值到底**从哪里来**，为什么没有反映真实输出。


## 2. 最终结论

先说结论，方便后面带着答案看过程：

1. `EUY` 指令的执行链路本身是通的。
2. `F_CMD_IO_OUTPUT = 200` 且 `type = 7` 时，主机会正确走到 `SetEUIOOutput()`，也就是**真的会让注塑机 EU 输出通断**。
3. 问题不在执行链路，而在**显示链路**。
4. HMI 的 `I/O -> EuOutput` 页面读取的是 `oStatus(7)`。
5. 但主机原来的周期状态上报代码没有把 `EUIO` 作为 `board 7` 上报给 HMI，因此 HMI 虽然会去读 `board 7`，却一直读不到更新后的值。
6. 已在主机侧补充 `board 7` 的周期状态上报，修复位置在：

`/home/harry/robot/EC-APRobotHost_/ec_develop/action/display.cpp`


## 3. 这次是怎么一步步找到问题的

这一段尽量用通俗方式说明排查思路。

### 第一步：先从 QML 入口看 `EUY` 是怎么产生动作对象的

用户先给了这一段：

`qml/App_HAMOUI/teach/OutputActionEditor.qml:704-707`

```qml
ICCheckBox{
    id:euY
    text: qsTr("EUY")
}
```

看到这里时，首先要回答的不是“它是什么”，而是：

- 点了它以后，当前编辑器切到了哪个 model？
- 选中的点位最后会被写成什么动作对象？

继续往上和往下读文件后，关键链路是：

1. 点击 `euY`
2. `typeGroup.onButtonClickedItem` 调用 `inside.handleCurrentGroupInfo(item)`
3. 当 `item == euY` 时，切换到 `euYModel`
4. 用户在网格里选择某个点后，`isSel` 被置为 `1`
5. `createActionObjects()` 遍历当前 model，把选中的项生成动作对象

对应到代码里：

- `qml/App_HAMOUI/teach/OutputActionEditor.qml`
  - `inside.handleCurrentGroupInfo()`
  - `createActionObjects()`
  - `onActionObjectChanged`

这一步的目的，是先确认：**EUY 并不是一个单独的特殊命令窗口，它最后还是被转成标准动作对象。**


### 第二步：去看动作对象定义，确认 `EUY` 最终对应什么 `type`

用户随后给了一个 `Reinstall` 目录下的同名文件片段，但 `Reinstall` 目录不参与当前项目实际运行，所以排查时应以项目内正在使用的文件为准：

`qml/App_HAMOUI/teach/extents/ExtentActionDefine.js`

这一段很关键，因为它定义了输出动作的标准格式。

从这里能确认：

- 动作码是 `action = 200`
- 核心字段有：`type`、`point`、`pointStatus`、`delay`、`isWaitInput`
- `generate()` 会输出标准动作对象

结合 `OutputActionEditor.qml` 中 `onActionObjectChanged` 的判断可以看出：

- `type == 7` 时，对应的就是 `euY`

也就是说，`EUY` 最终不是另外一种 action，而是：

```json
{
  "action": 200,
  "type": 7,
  "point": 某个EU输出点,
  "pointStatus": 0或1,
  "delay": 延时,
  "isWaitInput": false
}
```

这一步完成后，已经能把前端编辑器和动作协议接起来了。


### 第三步：再看协议定义，确认 `type = 7` 官方语义是什么

用户接着给了协议定义：

`vendor/protocol/hccommparagenericdef.h:1272-1278`

```cpp

 * 类型（EUIO，0~3:IO板，4～6：M值，7：EUIO；8：单头阀或者双头阀；9：停止检测；10：开始检测;）
 * IO点 当类型为8～10时候为阀ID

```

到这里，协议层就完全对上了：

- `action = 200` 就是 `F_CMD_IO_OUTPUT`
- `type = 7` 就是 `EUIO`

这一步的意义是：**前端推断出来的结论，和协议定义完全一致。**


### 第四步：转到主机代码，确认主机到底怎么执行 `F_CMD_IO_OUTPUT(type=7)`

接下来开始进入主机工程：

`/home/harry/robot/EC-APRobotHost_`

此时思路是这样的：

1. 先搜索 `F_CMD_IO_OUTPUT` 在主机里出现的位置。
2. 看看是谁负责解析这条指令。
3. 再看真正执行输出的是哪段代码。

- `ec_develop/action/ActionBase.cpp`
- `ec_develop/action/IOTask.cpp`

#### 4.1 解析动作对象

在 `IOOutput.h` 里可以看到：

- `Analytical()` 负责解析指令缓冲
- 读取出 `type`、`point`、`pointStatus`、`delay`、`isWaitInput`

说明主机确实把这条指令当作标准输出动作来处理。

#### 4.2 执行动作

在 `ActionBase.cpp` 里，`IOOutput::process()` 是关键。

其中这段逻辑最重要：

因为 `type = 7`，所以会进入 `type < 8` 分支，并且：

- `board_id1 = 7`
- `io_id1 = point`
- `io_status = on/off`

#### 4.3 最终打到硬件输出

继续顺着 `IOOperation()` 查到 `IOTask.cpp`：

```cpp

  switch (board) {
    case 7:  //< EUIO
    {
      SetEUIOOutput(id, on_off);
    } break;
```

再进 `SetEUIOOutput()`：

```cpp
void SetEUIOOutput(UINT32 id, int on_off) {
  if (on_off) {
    hc_euout.all |= 1 << id;
  } else {
    hc_euout.all &= ~(1 << id);
  }

  if (immboardType == 1)
    ioInterface->imm_easy.Outport(id, on_off);
  else if (immboardType == 2)
    ioInterface->imm_eu67.Outport(id, on_off);
}
```

到这里已经可以确定：

**EUY 确实会真正控制注塑机的 EU 输出。**

所以“有输出”这个现象不是错觉，而是执行链路本来就正确。


## 4. 为什么 `I/O` 页面里看不到 `EU输出`

当确认执行链路没问题后，思路就要切换：

> 既然输出真的发生了，那 HMI 监视页到底是看哪里的状态？

### 4.1 先找 I/O 监视页入口

从用户给的 `TopHeader.qml` 代码开始：

`qml/App_HAMOUI/TopHeader.qml:272-280`

```qml
TopMenuItem{
    id: io
    itemText: qsTr("I/O")
    onIsCheckedChanged: {
        parent.isCheckedHandlerHelper(io, isChecked);
    }
}
```

然后沿着这个入口继续找，定位到：

- `qml/App_HAMOUI/IOPage.qml`
- `qml/App_HAMOUI/IOComponent.qml`


### 4.2 `EuOutput` 页实际读取什么值

在 `qml/App_HAMOUI/IOComponent.qml` 的定时器刷新里，可以看到：

```qml
xStatus = [panelRobotController.iStatus(7)];
yStatus = [panelRobotController.oStatus(7)];
```

并且 `euyPages` 使用的就是这个 `yStatus`。

这一步说明：

- `EuInput` 页面读的是 `iStatus(7)`
- `EuOutput` 页面读的是 `oStatus(7)`

也就是说，HMI 逻辑其实很直接：

> 只要主机周期性把 `board 7` 的状态给过来，页面就能显示。


### 4.3 `oStatus(7)` 的数据来源

接着继续往 C++ 控制层找：

- `controller/panelrobotcontroller.h`
- `virtualhost/icrobotvirtualhost.h`
- `virtualhost/icrobotvirtualhost.cpp`

最后发现：

```cpp
Q_INVOKABLE quint32 oStatus(int boardID) const {
    return ICRobotVirtualhost::OStatus(boardID);
}
```

而 `ICRobotVirtualhost::OStatus(boardID)` 本质上就是从一个 map 里取值：

```cpp
static quint32 OStatus(int boardID) { return oStatusMap_.value(boardID, 0); }
```

再往下看 `oStatusMap_` 是怎么刷新的，发现：

```cpp
int boardID = HostStatusValue(&c_ro_5_3_0_938);
oStatusMap_.insert(boardID, HostStatusValue(&c_ro_0_32_0_940));
```

这说明 HMI 并不是直接去问“EU 输出当前多少”，而是等主机在**周期状态包**里告诉它：

- 当前这次上报的是哪个 board
- 这个 board 的输出状态是多少

然后 HMI 把这个值存进 `oStatusMap_[boardID]`。


## 5. 根因是怎么锁定到 `display.cpp` 的

现在问题就聚焦成一句话：

> 主机到底有没有把 `board 7` 的状态作为周期状态发回来？

这时候最合理的线索是找：

- `c_ro_5_3_0_938`
- `c_ro_0_32_0_940`
- 或者主机中负责周期上报 IO 状态的代码

最终定位到：

`/home/harry/robot/EC-APRobotHost_/ec_develop/action/display.cpp`

其中 `ECBoardIO(int board_num)` 函数就是负责周期性把 IO 状态塞到 `READ_PARA0` 结构里，发给 HMI。

### 5.1 原代码做了什么

原代码中，`ECBoardIO()` 会根据 `board_num` 和 `comm_cnt`，轮流上报：

- 普通 IO 板 `0 ~ 4`
- M 和附加板 `5 ~ 6`

但是没有看到任何地方上报：

- `io_id = 7`
- `io_in = GetEuIOin()`
- `io_out = GetEuIOout()`

### 5.2 这就解释了为什么“有输出但页面没显示”

因为主机内部其实已经维护了 EU 输出状态：

- `GetEuIOout()` 存在
- `SetEUIOOutput()` 也会更新 `hc_euout.all`

但是如果这个状态没有被 `display.cpp` 周期上报给 HMI，那么：

1. `I/O` 页面继续去读 `oStatus(7)`
2. `oStatusMap_[7]` 却始终没被更新
3. 页面当然就看不到真实变化

到这里，根因就算彻底闭环了。


## 6. 修复思路

修复思路很简单：

1. 在主机周期上报逻辑里补充 `board 7` 的 EU 输入/输出状态。
2. 让 HMI 收到 `io_id = 7` 的状态包。
3. `ICRobotVirtualhost` 收到以后自然会把值存入：
   - `iStatusMap_[7]`
   - `oStatusMap_[7]`
4. HMI 的 `EuInput` / `EuOutput` 页面就能正常显示。

实现上没有改 HMI，因为 HMI 的显示逻辑本来就是对的。

## 8. 为什么要这样改，而不是改 HMI

这个问题很容易误判成“是不是 HMI 页签有 bug”。

但排查后可以看出，HMI 侧其实没有明显错误：

1. `EuOutput` 页面就是去读 `oStatus(7)`
2. `EuInput` 页面就是去读 `iStatus(7)`
3. `euyDefines` 和 `euxDefines` 也都已经准备好了

也就是说，HMI 已经为 `board 7` 留好了显示位。

真正缺的是主机周期状态包里压根没有把 `board 7` 喂回来。

所以正确修复位置在主机，而不是 HMI。


## 9. 执行链路和显示链路的完整流程

### 9.1 执行链路

```text
用户在 OutputActionEditor 里选择 EUY
    -> 选择某个 euYModel 点位
    -> 生成 action=200, type=7 的动作对象
    -> 主机 IOOutput::Analytical() 解析该指令
    -> IOOutput::process() 根据 type=7 构造 board_id1=7
    -> IOOperation()
    -> SetIO(7, id, on_off)
    -> SetEUIOOutput(id, on_off)
    -> 注塑机 EU 输出硬件实际导通/断开
```

### 9.2 显示链路

```text
HMI 顶部进入 I/O 页面
    -> IOComponent.qml 定时刷新
    -> EuOutput 页面读取 panelRobotController.oStatus(7)
    -> oStatus(7) 实际来自 ICRobotVirtualhost::oStatusMap_[7]
    -> oStatusMap_[7] 的值来自主机周期状态包
    -> 主机 display.cpp 需要上报 io_id=7 的 io_out
```

### 9.3 原问题卡住的地方

```text
EUY 已经真实输出
    -> 主机内部 GetEuIOout() 也有值
    -> 但 display.cpp 没有把 board 7 周期上报给 HMI
    -> oStatusMap_[7] 没更新
    -> EuOutput 页面看不到变化
```

### 2. 注塑信号灯页面

新增的注塑信号灯页面与 I/O 监视页不完全相同：

- `开模完 / 可关模 / 可顶进 / 再循环` 是按注塑工艺约定读取固定 EUIO 点。
- `安全门` 是按系统 `safeDoor` 配置解释后的逻辑状态读取。

这样设计的目的：

- 把“原始 I/O 状态显示”和“业务语义状态显示”分开。
- 对操作员来说，`安全门` 更应该表示“系统当前认定的安全门状态”，而不是单纯某一根线的原始电平。
