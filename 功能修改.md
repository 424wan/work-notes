# Easy Program 检测项改动复盘

## 1. 问题起点 / 需求背景

### 最初的问题或需求是什么

这次需求不是单纯改几个按钮文字，而是要把“单个确认限检测”这套逻辑从旧的“通/断检测模型”改成新的“8 个独立检测项模型”。

用户给出的核心要求有两层：

- 本页面不再自己区分“通检测 / 断检测”对方向的影响。
- 最终插入指令的 `xDir`，统一只由 `EasyProgramSignalPage.qml` 里的“正向 / 反向”配置决定。

同时，用户给了新的对应关系：

- 吸1通检测 -> 吸1检测
- 吸1断检测 -> 夹1检测
- 吸2通检测 -> 吸2检测
- 吸2断检测 -> 夹2检测
- 夹1通检测 -> 吸3检测
- 夹1断检测 -> 夹3检测
- 夹2通检测 -> 吸4检测
- 夹2断检测 -> 夹4检测

### 初始现象或期望效果

最初看起来像是“检测项和方向配置的映射要调整”，但真正的期望效果其实更深：

- 页面上的检测按钮名称，要和新的检测配置项一致。
- 插入的检测指令，方向不能再被“通 / 断”二次处理。
- 指令回填、模板快设、动作显示文本，也都要跟着改，否则 UI 和底层动作会不一致。

### 当时最可能的几个原因或实现思路

一开始最可能的思路有 3 个：

1. 只改 `signalKey` 映射就够了。
   原因是用户最先给的是“吸1通检测 -> 吸1检测”这一类映射，看起来像只是改方向来源。

2. 不只是映射问题，旧的“通 / 断反转方向”逻辑也要删。
   因为用户明确说“本页面不做通断判断了，xDir 只靠 SignalPage 的正反向设置”。

3. 还可能存在更深的结构问题：按钮显示名、动作回填、底层点位 `pointName` 可能仍然绑定着旧语义。
   这个判断来自经验：UI 配置项经常不只是显示用途，而是串到动作创建和动作回填里。

## 2. 分析与排查过程

### 步骤 1

#### 当时的判断是什么

先确认这次需求到底影响哪一层：只是按钮显示，还是会影响最终插入动作。

#### 为什么这么判断

如果只是改显示名，改 `name` 就行；如果会影响动作，就必须继续追 `checkSignalRows` 这组数据到底用在什么地方。

#### 做了什么验证

- 查看 `EasyProgramFunctionPanel.qml` 里 `checkSignalRows`
- 查看检测面板的 UI 绑定
- 搜索 `signalKey / isOnCheck / preferEnd / checkDirectionProvider` 的使用位置

#### 结果如何

很快确认 `checkSignalRows` 不是单纯 UI 数据，它直接参与：

- 检测动作创建
- 已有动作回填
- 检测方向归一化

这说明不能只改显示。

### 步骤 2

#### 当时的判断是什么

需要搞清楚 `EasyProgramSignalPage.qml` 的“正向 / 反向”配置到底怎么接入当前页面。

#### 为什么这么判断

用户明确说 `xDir` 只靠 SignalPage 控制。如果不知道这条链路怎么接，就没法判断应该改哪一层。

#### 做了什么验证

- 查看 `EasyProgramSignalPage.qml` 里的 `getCheckDirectionValue`
- 查看 `checkDirectionKeys` 和 8 个配置项
- 搜索 `checkDirectionProvider` 是谁传进来的

#### 结果如何

确认链路是：

- `EasyProgramSignalPage.qml` 提供 8 个检测方向配置
- `EasyProgramWizard.qml` 把 `signalPage` 传给 `quickPage`
- `EasyProgramQuickPage.qml` 再传给 `EasyProgramFunctionPanel.qml`

这一步非常重要，因为它证明“方向来源”本来就已经是 SignalPage，只是中间还有旧逻辑在做二次加工。

### 步骤 3

#### 当时的判断是什么

旧逻辑里肯定存在“通 / 断导致方向反转”的分支。

#### 为什么这么判断

用户已经明确说“不做通断判断”，而旧 UI 名称正好是“吸1通检测 / 吸1断检测”这类结构，这通常意味着程序会基于通断做特殊处理。

#### 做了什么验证

- 查看 `getCheckRowDirectionValue(checkRow)`
- 查看 `actionCheckOnState(actionObject, pointName)`
- 查看 `presetCheckDirectionValue(preset)`

#### 结果如何

找到 3 个关键旧逻辑：

1. `EasyProgramFunctionPanel.qml` 里，`isOnCheck === false` 时会反转方向。
2. `findCheckRowByAction()` 会根据 `checkOnState` 或 `xDir` 推断“通 / 断行”。
3. `EasyProgramQuickPage.qml` 的模板预设也会按 `isOnCheck` 反转方向。

这一步确认了第一轮改动方向：不是只改映射，而是要删掉“断检测自动反向”的逻辑。

### 步骤 4

#### 当时的判断是什么

第一轮可以先改成“方向直接看 signalKey”，再观察还有哪些不一致。

#### 为什么这么判断

这是最直接的主链路修复。先解决用户最明确指出的问题，再看剩余问题是否暴露。

#### 做了什么验证

- 修改 `checkSignalRows` 的 `signalKey`
- 删除 `getCheckRowDirectionValue()` 里的反转逻辑
- 修改模板预设方向逻辑
- 检查回填和规范化逻辑是否还能工作

#### 结果如何

主方向链路改通了，但很快暴露出新问题：

- `name` 还是旧文案
- 回填逻辑虽然能工作，但仍然依赖旧的“通 / 断”结构

这说明问题比“方向逻辑”更深，数据模型本身还没切换完成。

### 步骤 5

#### 当时的判断是什么

按钮显示名 `name` 也必须同步改，否则页面语义和 SignalPage 的检测项不一致。

#### 为什么这么判断

用户直接指出“你没有改 name”，这说明从用户视角看，页面名称本身就是需求的一部分，而不仅仅是底层动作。

#### 做了什么验证

- 对照用户给的 8 组映射
- 修改 `checkSignalRows` 的 `name`
- 再次检查 `displayText` 是否引用 `checkRow.name`

#### 结果如何

显示名问题解决了，但这一步也触发了更关键的思考：

如果名称已经变成“吸3检测 / 夹3检测”，那原来共用的 `pointName` 还合理吗？

### 步骤 6

#### 当时的判断是什么

必须重新检查 `pointName` 的真实含义，不能假设它只是附带信息。

#### 为什么这么判断

这是一个典型的风险点：很多时候开发者只看到了“按钮映射”，没继续追“动作最终到底写入了什么”。如果 `pointName` 真参与动作生成，那不改就会造成“界面看着对，动作点位错”。

#### 做了什么验证

- 查 `createCheckQuickAction()` 是否使用 `pointName`
- 查 `findCheckRowByAction()` 是否依赖 `pointName`
- 查 `SpecialIODefines.js` 里 X010~X017 的真实定义

#### 结果如何

确认了两件最关键的事实：

1. `pointName` 直接决定最终插入动作的 `point`
2. X 点定义真实对应关系是：
   - X010~X013 -> 夹1~夹4确认限
   - X014~X017 -> 吸1~吸4确认限

这一步让问题真正升级成“数据模型错了”，而不再是“逻辑细节没改完”。

### 步骤 7

#### 当时的判断是什么

既然已经切成“8 个独立检测项”，那旧字段 `preferEnd / isOnCheck / checkOnState` 基本都不应该继续留在主流程里。

#### 为什么这么判断

这三个字段都服务于旧模型：

- 同一个 `pointName` 有两行
- 两行靠“通 / 断”区分
- 回填时需要额外信息定位是哪一行

但新模型里每一行都应该是独立的：

- 一个独立名称
- 一个独立点位
- 一个独立方向配置来源

这种情况下继续保留旧字段，只会让逻辑更绕、更容易出错。

#### 做了什么验证

- 搜索 `preferEnd / isOnCheck / checkOnState` 的全部引用
- 判断这些引用是在主流程中必要，还是仅用于旧兼容

#### 结果如何

确认可以把主流程彻底切成：

- `name` 负责显示
- `pointName` 负责动作点位
- `signalKey` 负责方向来源

旧字段可以移除。

### 步骤 8

#### 当时的判断是什么

最终方案应该是彻底重构 `checkSignalRows` 的含义，而不是在旧模型上打补丁。

#### 为什么这么判断

如果只是继续修补旧字段，会出现一个非常典型的问题：

- 页面名称是新语义
- 点位还是旧语义
- 回填逻辑又夹着旧兼容

这种状态最危险，因为“每一层都看起来能跑”，但整体已经不一致了。

#### 做了什么验证

最终进行了几类改动：

- 把 8 个 `pointName` 改成真实 X010~X017 对应
- 删除 `preferEnd / isOnCheck / checkOnState` 在主流程中的作用
- 回填逻辑改成优先按 `point + signalKey` 识别
- 保留 `checkSignalKey / checkDisplayName`，用于动作回填和动作显示

#### 结果如何

最终模型被收敛成“8 个独立检测项”，和需求一致。

## 3. 过程中出现的新问题

### 新问题 1：只改了 `signalKey`，没改 `name`

#### 出现原因

第一轮过度聚焦“方向来源”，忽略了页面显示语义本身也是需求的一部分。

#### 问题本质属于

UI 问题

### 新问题 2：`name` 改了，但 `pointName` 还是旧映射

#### 出现原因

最开始把 `pointName` 当成了“辅助字段”，没有第一时间意识到它直接参与动作 `point` 的生成。

#### 问题本质属于

逻辑问题

### 新问题 3：新语义已经是“8 个独立检测项”，但回填逻辑还保留旧“通 / 断”模型

#### 出现原因

旧系统原本是一组点位对应两行按钮，所以回填依赖 `preferEnd / isOnCheck / checkOnState`。第一轮修改只改了方向，没有同步重构行身份识别方式。

#### 问题本质属于

状态问题 + 逻辑问题

### 新问题 4：动作显示文本可能继续沿用旧点位语义

#### 出现原因

动作字符串不是直接读按钮文本，而是有自己的 `toStringHandler`。如果不一起处理，会出现“面板选中对了，但动作描述还像旧逻辑”的情况。

#### 问题本质属于

UI 问题 + 逻辑问题

## 4. 关键转折点

### 哪一步让问题真正接近解决

真正的关键转折点，是去查 `pointName` 和 X010~X017 的底层定义那一步。

### 为什么这一点是关键

因为在那之前，整个问题都还像是“方向逻辑怎么改”的问题；查完之后才发现，本质上是“数据模型已经变了，但代码还按旧模型组织”。

换句话说：

- 之前解决的是“怎么让方向看起来对”
- 查完 `pointName` 后，开始解决“怎么让动作本身真正对”

### 如果没有这一步，会一直卡在哪

会一直卡在一种很危险的假修复状态：

- 页面名称看起来对
- 方向也看起来对
- 但底层点位写错

这种问题最难发现，因为表面上“像是已经修好了”。

## 5. 最终解决方案

### 最终是怎么修复的

核心思路不是继续修补旧“通 / 断检测模型”，而是直接切成“8 个独立检测项模型”。

也就是把每个检测项明确成 3 个独立维度：

- 这项叫什么
- 它对应哪个真实确认限点位
- 它的方向从哪个 SignalPage 配置项取

### 涉及的关键点

1. `checkSignalRows` 只保留 `name / pointName / signalKey`，并改成真实 X010~X017 对应。
2. `xDir` 不再根据“通 / 断”反转，而只看 `EasyProgramSignalPage.qml` 的正反向配置。
3. 回填和显示文本统一改为基于新模型识别，去掉旧 `preferEnd / isOnCheck / checkOnState` 主逻辑。

## 6. 如果重新来一遍，最快路径是什么

### 可以跳过哪些无效尝试

可以跳过这类“局部修补”：

- 只改 `signalKey`
- 只改 `name`
- 继续保留旧“通 / 断”结构，再往上叠补丁

这些尝试不是完全没价值，但会拖慢收敛速度。

### 最优排查顺序应该是怎样

最优顺序应该是：

1. 先确认 `checkSignalRows` 每个字段分别被谁使用
2. 再确认 `xDir` 的真实来源链路
3. 再确认 `pointName` 是否参与最终动作生成
4. 最后判断旧字段是不是还能保留

这条路径的好处是：先把“字段语义”搞清楚，再动代码，不容易出现改一半发现模型不对的情况。

## 7. 本质总结

### 这个问题的本质原因一句话总结

这个问题的本质原因是：需求已经把“通 / 断双态模型”改成了“8 个独立检测项模型”，但代码最初只改了映射，没有同步重构数据模型和点位语义。

### 以后遇到类似问题，可以优先检查什么

以后遇到这种“配置项映射变更”问题，优先检查这 4 件事：

1. 这个字段是不是只影响显示，还是会影响最终动作 / 数据落库
2. 同一组配置数据，是否同时参与了创建、回填、显示三个环节
3. 页面名称变化后，底层点位和状态字段是否还沿用旧语义
4. 当前代码是不是在拿“兼容旧模型的字段”继续驱动新模型

经验上，凡是“看起来只是改映射”的需求，都要多问一句：

这个映射背后，改的是显示层，还是改的是整套数据模型。


# 等待开模完问题调试复盘

## 1. 问题起点 / 需求背景

### 最初的问题或需求是什么

这次的目标不是单纯“改一个等待指令”，而是要把简易编程里的“等待开模完”做成一种受限行为：

- 信号固定为 `EuX013:开模完`
- 触发方式固定为“上升沿”
- `不限时` 保留，但置灰不可勾选
- `确认时间` 保留，但置灰不可编辑
- 用户只允许填写“限时 / 等待时间”
- 插入时要生成 `910`，也就是 `injectWaitSafedoorAction`

### 初始现象或期望实现的功能效果

一开始代码里已经存在一个 `910` 指令定义，但简易编程里的“等待”功能并没有直接走它。

实际现象是：

- 注塑工艺模板里的“等开模完”在其他地方会生成 `910`
- 但简易编程等待面板本身仍然是通用等待逻辑
- 用户可以自由选信号、选触发方式、勾不限时、改确认时间

所以问题不是“没有这个指令”，而是“同一个业务含义，在不同入口里走了不同实现”。

### 当时最可能的几个原因或实现思路

最开始我脑子里主要有 3 条思路：

1. 可能“等待开模完”本来就有专门入口，只是我还没找到调用链。
2. 可能需要新做一个专用 UI，再把它单独绑定到 `910`。
3. 也可能根本不需要新界面，只要在现有“等待”里识别 `EuX013` 这个组合，局部锁死选项即可。


## 2. 分析与排查过程

### 步骤 1

#### 当时的判断是什么

先确认“这个指令”到底是谁在调用，不能直接假设简易编程已经在用 `910`。

#### 为什么这么判断

因为用户先问的是 `ExtentActionDefine.js` 里那段定义是谁调用的。  
如果调用链都没弄清楚，后面改 UI 很容易改错地方。

#### 做了什么验证

- 看了 `ExtentActionDefine.js` 里 `injectWaitSafedoorAction` 的定义
- 全局搜索 `injectWaitSafedoorAction`
- 全局搜索动作码 `910`

#### 结果如何

确认了两件事：

- 注塑工艺生成流程里，`InjectTechnology.qml` 会调用 `injectWaitSafedoorAction.generateActionObject(...)`
- 协议里 `910` 的常量名是 `F_CMD_INJECT_WAIT_MODE_OPEN`

这一步得到的结论很关键：  
`910` 不是新东西，系统里早就有，只是要看简易编程有没有用它。


### 步骤 2

#### 当时的判断是什么

接着往简易编程里追，“等开模完”这个槽位到底现在生成的是什么。

#### 为什么这么判断

用户需求里明确提到“插入时生成指令也需要调用刚才的 910，现在又是调用的什么也说明一下”。  
这句话的重点不是 UI，而是“当前实现路径”。

#### 做了什么验证

- 查了 `EasyProgramQuickPage.qml` 里“等开模完”的模板预设
- 查了 `EasyProgramFunctionPanel.qml` 里等待面板的回填和创建逻辑
- 重点看了 `createWaitQuickAction()` 和 `buildPresetWaitActionObject()`

#### 结果如何

确认当前简易编程并没有统一走 `910`：

- 模板预设里的“等开模完”原本是普通 `wait`
- 等待面板插入逻辑原本走的是 `extentWaitInputAction.generateActionObject(...)`

也就是说，业务名字叫“等待开模完”，但实现其实还是“通用等待输入”。


### 步骤 3

#### 当时的判断是什么

第一版我判断：为了只影响模板里的“等开模完”，可以给它单独加一个 preset 类型，再把等待面板切成专用受限模式。

#### 为什么这么判断

当时我担心一个问题：  
如果直接在现有等待里写死 `EuX013 -> 910`，会不会误伤其他普通等待场景。

所以我选择了“隔离式改法”：

- 新增一个专用 preset 标记
- 把这个标记传到面板
- 面板根据标记显示受限 UI

#### 做了什么验证

- 修改了模板预设类型
- 给功能面板传额外上下文
- 在等待面板里增加了一组专用灰色展示控件
- 插入逻辑按这个专用模式直接生成 `910`

#### 结果如何

功能上是能工作的大方向，但很快暴露出一个问题：  
这不是最小改动，而且引入了额外结构，复杂度明显偏高。


### 步骤 4

#### 当时的判断是什么

用户明确指出“这不是最小改动”，我重新判断后认为用户是对的，应该退回到现有等待界面里做条件限制，而不是做一套专用界面逻辑。

#### 为什么这么判断

需求本身其实只依赖一个条件：

- 当等待类型是“塑机信号”
- 且信号是 `EuX013`

这时把无关项置灰，并改走 `910` 就够了。  
这个条件本来就能在现有控件上判断，不需要新概念。

#### 做了什么验证

- 删除了新增的 preset 特化路径
- 删除了给面板传专用上下文的逻辑
- 保留原等待面板
- 改成在现有面板内通过 `waitMachineSignalBox + waitMachineSignalSelector.currentText === EuX013` 判断是否进入受限态

#### 结果如何

代码规模明显缩小，结构也更贴近原设计。  
这一轮之后，方向才真正回到“最小改动”。


### 步骤 5

#### 当时的判断是什么

既然不新增界面，那就应该只改“现有控件的状态”，不要再额外摆一组灰色控件。

#### 为什么这么判断

“置灰不可改”不等于“再画一份灰色版本”。  
最小改动的本质是：

- 保留现有控件
- 只改它们的 `enabled / checked / opacity`

这样才是真正沿用原界面结构。

#### 做了什么验证

- 删除了额外新增的灰色展示行
- 直接在原来的 `断 / 通 / 上升沿 / 下降沿` 上做禁用
- 直接在原来的 `不限时 / 确认时间` 上做禁用
- 保留原来的塑机信号下拉框

#### 结果如何

界面结构回到了原样，视觉上只是“同一套控件在 `EuX013` 时进入锁定态”。


### 步骤 6

#### 当时的判断是什么

虽然大方向对了，但后来用户反馈：“塑机信号现在只能选择开模完了，其他都不能选了。”

这说明我在锁定 `EuX013` 功能时，把“默认化”写成了“强制回写选择”。

#### 为什么这么判断

如果用户一改下拉框，界面又自动跳回 `EuX013`，通常只有一种原因：

- 在 `onCurrentIndexChanged` 或类似联动里写了“只要满足某条件，就把 `currentIndex` 改回第四项”

这不是 UI 样式问题，而是典型的状态回写问题。

#### 做了什么验证

我重新看了等待面板相关代码，定位到：

- `applyInjectWaitModeOpenDefaults()` 里不仅锁状态，还会把下拉框重置到 `EuX013`
- 而这个函数又挂在了下拉框 `onCurrentIndexChanged` 上

#### 结果如何

根因确认了：  
不是“下拉框被禁用”，而是“状态修正函数顺带改了下拉框本身”，造成用户无法离开第四项。


### 步骤 7

#### 当时的判断是什么

必须把“锁定行为参数”和“回填已有 910 动作时选中 EuX013”拆成两个函数。

#### 为什么这么判断

这两个动作虽然都和 `EuX013` 有关，但本质不同：

- 一个是“普通用户手动选择后，界面该如何受限”
- 一个是“编辑一个已经存在的 `910` 动作时，界面该如何回填”

如果继续混用同一个函数，就会反复出现“用户改状态时被程序写回”的问题。

#### 做了什么验证

- 保留 `applyInjectWaitModeOpenDefaults()`，只做状态锁定
- 新增 `applyInjectWaitModeOpenActionDefaults()`，只在回填 `910` 时才把下拉框切到 `EuX013`
- 让下拉框的 `onCurrentIndexChanged` 只调用前者，不再改下拉框本身

#### 结果如何

问题解决：

- 其他塑机信号可以正常选
- 只有用户真的选到 `EuX013` 时，才会锁住其他参数
- 已有 `910` 动作在编辑时仍能正确回填成 `EuX013`


## 3. 过程中出现的新问题

### 新问题 1：改动过大，不是最小方案

#### 出现原因

一开始为了避免误伤普通等待场景，我引入了“专用 preset + 专用受限界面”的思路。  
这在功能上可行，但违背了“尽量沿用现有结构”的目标。

#### 本质分类

- 逻辑问题
- UI 问题
- 设计收敛问题


### 新问题 2：新增了额外一组灰色 UI

#### 出现原因

我把“置灰不可改”理解成了“新画一套灰色展示控件”，而不是直接禁用现有控件。  
这会让界面结构变复杂，也更不符合最小改动原则。

#### 本质分类

- UI 问题


### 新问题 3：塑机信号被锁死，只能选开模完

#### 出现原因

这是这次修改里最典型的副作用。  
根因不是控件本身，而是“状态修正函数做了两件事”：

- 锁定其他参数
- 强制把下拉框改回 `EuX013`

再加上这个函数被放进了下拉框变化事件里，于是用户每次尝试切换，都会被系统拉回去。

#### 本质分类

- 状态问题
- UI 交互问题


## 4. 关键转折点

### 哪一步让问题真正接近解决

真正的关键转折点有两个：

1. 从“新增专用界面”退回到“在现有等待界面里做条件限制”
2. 把“锁状态”和“回填下拉框”拆成两个函数

### 为什么这一点是关键

第一点解决的是“方向错了”的问题。  
第二点解决的是“方向对了，但状态写坏了”的问题。

如果没有第一点，会一直在多余的结构上打补丁。  
如果没有第二点，会一直卡在“看起来功能正常，但用户一操作就被强制改回去”的交互 bug 上。

### 如果没有这一步，会一直卡在哪

- 没有第一点：会一直在讨论“要不要专用界面”，代码会越来越绕
- 没有第二点：`EuX013` 的功能虽然能用，但其他塑机信号永远选不正常


## 5. 最终解决方案

### 最终是怎么修复的

最终方案非常简单：

- 继续使用原来的等待面板
- 在“塑机信号 + EuX013”这个组合下，把无关项锁定
- 插入时对这个组合单独改走 `910`

### 涉及哪几个关键点

1. 识别条件只看当前选择是否为“塑机信号 + EuX013”
2. 受限态只修改现有控件的状态，不增加新界面
3. 生成动作时对 `EuX013` 单独分支到 `injectWaitSafedoorAction`


## 6. 如果重新来一遍，最快路径是什么

### 可以跳过哪些无效尝试

可以直接跳过这些尝试：

- 新增一种专用模板类型
- 额外传一套上下文给等待面板
- 再画一组灰色展示控件

这些都不是解决这个问题的最短路径。

### 最优排查顺序应该是怎样

最优顺序应该是：

1. 先确认当前“等开模完”到底生成的是什么指令
2. 再确认 `910` 在别的入口里是怎么生成的
3. 最后只在现有等待面板里加一个条件判断：
   当选择 `EuX013` 时锁定参数并生成 `910`

这样一步到位，既不会误判调用链，也不会把问题做复杂。


## 7. 本质总结

### 这个问题的“本质原因”一句话总结

本质上，这是一个“同一个业务语义在不同入口里实现不一致，而且状态锁定逻辑和回填逻辑混在一起”的问题。

### 以后遇到类似问题，可以优先检查什么

以后遇到这类问题，优先检查两件事：

1. 这个业务在系统里是不是已经有专用指令，只是当前入口没用上
2. 你的“状态修正函数”有没有顺手改了用户当前选择，导致交互被锁死

再说得更直白一点：

- 先分清“生成什么”
- 再分清“界面怎么限制”
- 最后分清“回填已有数据”和“用户手动操作”是不是用了同一套状态逻辑

这三件事只要混在一起，问题通常就会越改越绕。
