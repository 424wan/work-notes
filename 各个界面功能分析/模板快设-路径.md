# 姿势直线、自由路径按键组
这组按键本质上是一个“路径类型单选组”，代码在 EasyProgramFunctionPanel.qml (line 1176) 到 EasyProgramFunctionPanel.qml (line 1208)。

它的结构很直接：
	- 外层是 ICButtonGroup，id 是 pathTypeSel
	- 里面有两个 ICCheckBox
	- poseLine3DType 对应“姿势直线”
	- freePathType 对应“自由路径”

这组按钮的核心作用，是把 UI 里的“选中哪个路径类型”映射到 root.pathTypeIndex：
	- 0 表示“姿势直线”，见 EasyProgramFunctionPanel.qml (line 1193)
	- 1 表示“自由路径”，见 EasyProgramFunctionPanel.qml (line 1201)

这几个点最关键：
1. mustChecked: true  
    表示必须始终有一个被选中，不能两个都不选。这个行为是在 ICButtonGroup.qml (line 8) 和 ICButtonGroup.qml (line 55) 实现的。

2. checkedIndex: root.pathTypeIndex  
    组的当前选中项和外部状态 root.pathTypeIndex 对齐，说明真正的业务状态不保存在按钮内部，而是保存在 root 上，按钮组只是视图层入口。

3. 子按钮的 isChecked 也绑定了 root.pathTypeIndex  
    在 EasyProgramFunctionPanel.qml (line 1198) 和 EasyProgramFunctionPanel.qml (line 1206)，实际上又做了一层“索引 -> 勾选状态”的映射。  
    也就是说：
		- pathTypeIndex === 0 时，第一个框勾上
		- pathTypeIndex === 1 时，第二个框勾上

4. 点击后会同步到底层路径动作  
    这组按钮不只是切换显示，它会调用 root.syncPathEditorAction()，见 EasyProgramFunctionPanel.qml (line 1184) 和 EasyProgramFunctionPanel.qml (line 1188)。  
    这个函数最终会把当前 UI 选项转换成具体动作类型，并设置给 pathEditor，见 EasyProgramFunctionPanel.qml (line 445) 和 EasyProgramFunctionPanel.qml (line 510)。

结合 pathActionFromUi() 来看，这组按钮控制的是：

- 选“姿势直线”时：
	- 非相对：F_CMD_LINE3D_MOVE_POSE
	- 相对：F_CMD_LINE_RELATIVE_POSE

- 选“自由路径”时：
	- 非相对：F_CMD_JOINTCOORDINATE
	- 相对：F_CMD_JOINT_RELATIVE

也就是它和旁边那个“相对”复选框一起，决定 pathEditor 当前编辑的是哪一种路径动作。

还有一个实现层面的特点值得注意：
	- onButtonClickedID
	- onCheckedIndexChanged

这两个回调里都做了：
	- root.pathTypeIndex = ...
	- root.syncPathEditorAction()

所以一次正常点击，很可能会触发两次同步。功能上通常没问题，但从代码设计上看有点重复，属于“能跑但略冗余”的写法。

## 但从代码设计上看有点重复，属于“能跑但略冗余”的写法
意思是“同一次点击”，代码里有两条路都会去调用同一个同步函数。

以你点“自由路径”为例，流程大概是这样：
1. 你点了 ICCheckBox
2. 它的 isChecked 变成 true
3. ICButtonGroup 监听到这个变化后，先把 checkedIndex = 1
4. checkedIndex 一变，就触发了这段：
```qml EasyProgramFunctionPanel.qml (line 1188)
onCheckedIndexChanged: {
    root.pathTypeIndex = checkedIndex;
    root.syncPathEditorAction();
}
```
5. 接着 ICButtonGroup 又主动发了 buttonClickedID(index) 信号
6. 于是又触发这段：
```qml EasyProgramFunctionPanel.qml (line 1184)
onButtonClickedID: {
    root.pathTypeIndex = index;
    root.syncPathEditorAction();
}
```
所以结果就是：
- 第一次：因为 checkedIndex 变了，调用一次 syncPathEditorAction()
- 第二次：因为 buttonClickedID 信号发了，又调用一次 syncPathEditorAction()

我说“冗余”，就是指这两次做的事几乎一样，最终都会把 pathTypeIndex 设成同一个值，再把动作同步给 pathEditor。

这不一定会出 bug，因为两次结果一致，只是：
- 多做了一遍相同工作
- 读代码时容易让人迷惑
- 如果以后 syncPathEditorAction() 变重了，可能会有额外副作用或性能浪费

更直白一点说，就是：
“点一次按钮，业务上只需要通知一次就够了，但现在代码可能通知了两次。”

## 为何比InjectTechnology.qml改动代码多
有差异，而且差异还挺关键，所以不能机械照抄。

InjectTechnology.qml 这组按钮其实是“自己管自己”的写法，见 InjectTechnology.qml (line 944)：
- 它自己额外维护了两个状态：path 和 which
- 真正的选中态是 which
- 点按钮时先弹框，确认后才改 which
- 两个 ICCheckBox 的 onIsCheckedChanged 又会把 isChecked 强行拉回到 which 对应的状态

也就是说，它本来就留了一个“中间态”，方便做“先确认，再切换”。

但 EasyProgramFunctionPanel.qml 原来不是这个结构，见 EasyProgramFunctionPanel.qml (line 1236)：
- 它直接用 ICButtonGroup.checkedIndex
- 业务状态直接就是 root.pathTypeIndex
- 组内切换时会立刻进 onCheckedIndexChanged
- 然后马上 syncPathEditorAction()

也就是说，它原来是“点了就切”，没有 InjectTechnology 那种专门拿来回退的 which 状态。

还有第二个差异，清空内容也不一样：
- InjectTechnology 清空的是 6 个输入框，见 InjectTechnology.qml (line 985)
- EasyProgramFunctionPanel 里“记录位置”在 pathEditor.points[0] 里，实际是另一个组件的数据，不是几个直接摆在当前文件里的编辑框，见 EasyProgramPathEdit.qml (line 145) 和 EasyProgramPathEdit.qml (line 163)

所以我这次改动看起来比它多，主要是为了补这几个能力：
- 先拦截按钮组切换
- 弹框后还能恢复原来的选中状态
- 确认后清掉 pathEditor 里的已记录点位
- 不把原来的 pathTypeIndex -> pathEditor action 逻辑打散


## 为什么要拦截按键组的切换，现在的现象是我点击切换的时候，选择否，他无法进行切换

要拦截，是因为这个 ICButtonGroup/ICCheckBox 不是“点完再通知”，而是“点下去就先把选中状态改掉”。如果不拦截：
- 你一点击，按钮组已经切到新类型了
- pathTypeIndex 和 pathEditor 的动作也会立刻跟着变
- 这时候再弹“是否清空当前记录位置”，这个弹框就只能做提示，没法真正决定这次切换怎么落地

所以这里拦截的目的其实不是“阻止切换”，而是把切换延后到用户选完弹框之后再执行。

你说的现象也是我这里逻辑定错了：我之前把“否”写成了“取消切换”。但按这句文案更合理的语义应该是：
- 是：清空当前记录位置，然后切换
- 否：不清空当前记录位置，但仍然切换

这个我已经改了，位置在 EasyProgramFunctionPanel.qml (line 564)。现在 否 会走 applyPathTypeSelection(targetIndex, false)，不会再卡住不切换。
