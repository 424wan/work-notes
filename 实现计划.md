# 注塑工艺专用 InjectSpeedControlArea 改版方案
## Summary
在 TechnologyType == 6 下，injectSpeedControl 保持现有弹出尺寸不变，但将 InjectSpeedControlArea.qml 重构为一页左右平分的注塑工艺专用界面。左侧复用 SpeedControlArea 的速度/寸动逻辑，右侧复用 ArmMovePage 的世界坐标、关节坐标和 partkeyboardSection 点动逻辑。main.qml 仅保持当前“注塑工艺时打开 injectSpeedControl、隐藏 armKeyboardContainer”的接线，不新增新的切换规则。

## Key Changes
1. InjectSpeedControlArea.qml 重新组织为左右 1:1 布局
    - 保持当前根组件尺寸与关闭方式不变，继续对外保留 SpeedControlArea 已在用的接口：inchEnableChange、keyType、reflashUI(speed)、onKeyInching(key)。
    - 根内容改成左右平分：
        - 左侧是速度/寸动设置区。
        - 右侧是简化版手动点动区。
    - 保留底部“关闭”按钮，位置按图片在整体底部中间。
2. 左侧速度区复用 SpeedControlArea 现有逻辑，但重排布局
    - 上半区：
        - 复用现有 currentSpeed 作为“手动速度”输入控件，保留原百分比输入和勾选行为。
        - 将现有纵向速度滑条 + speedTextArea 改为横向表现；底层仍继续使用 setRealSpeed() / getyhBySpeed() / reflashUI() 维护速度值、滑块位置、全局速度和控制器写入。
        - 新增 3 个固定手动速度快捷按钮：1%、10%、30%。点击后直接调用 setRealSpeed()，同时同步 currentSpeed、滑条和全局速度显示。
        - 原 quickClickSpeed / clearAllQucikClick / save / customSpeedModel / quickSpeedClickArea 这一整套“自定义速度列表”从注塑页移除。
    - 下半区：
        - 复用现有 inchBindKey 作为“寸动距离”输入控件，保留原勾选和写控制器逻辑。
        - 新增 2 个固定寸动距离快捷按钮：0.1mm、1mm。点击后统一走一个新的内部 helper（例如 applyInchDistance(value)），完成：
            - 更新 inchBindKey.configValue
            - 写 s_rw_1_16_3_260
            - panelRobotController.syncConfigs()
        - 原 currentClickSpeed、bindExternKey、pSpeedUp/pSpeedDown、cSpeedUp/cSpeedDown、点击速度快捷设置按钮从注塑页移除。
    - 默认值沿用当前 SpeedControlArea 初始化逻辑，不新增新的配置项。
3. 右侧点动区复用 ArmMovePage 的世界/关节 + 点动功能
    - 仅保留 3 个复选框：
        - worldcodinate
        - jointcodinate
        - 新增 lockPosture
    - worldcodinate / jointcodinate 的样式与功能直接沿用 ArmMovePage.qml 现有逻辑：
        - 世界坐标：设置 ShareData.codinateType = ShareData.worldcodinate，写 Keymap.COMBINE_ARM_MOVE_TYPE
        - 关节坐标：设置 ShareData.codinateType = ShareData.jointcodinate，写 Keymap.SINGLE_ARM_MOVE_TYPE
    - 下方直接放 partkeyboardSection，保留原按钮文本、正负方向命令、点动/寸动分支、AxisDefine 可见性掩码逻辑。
    - 不引入以下内容到注塑页：
        - toolCodinate
        - userCodinate
        - coordSel / toolSel / extoolSel
        - tuneEn / tuneSel / tuneSpeedGroup / keyboardSectionToSel
        - 任何外部工具、用户坐标、手轮轴选择 UI
    - lockPosture 这版只做 UI：
        - 有独立 ICCheckBox
        - 维护本页内部勾选状态
        - 不接控制器、不限制按钮、不改变坐标/点动行为
4. main.qml 只保持现有接线，不扩展新职责
    - 继续使用当前 isInjectTechnology() 与 currentSpeedControl() 作为选择入口，不再新增额外入口函数。
    - onConfigClicked 维持现状：注塑工艺时打开 injectSpeedControl，其他工艺打开 speedControl。
    - armKeyboardContainer.visible 维持现状：注塑工艺下隐藏，其他工艺在手动模式下显示。
    - injectSpeedControl.inchEnableChange.connect(mainHeader.setModeText) 保留，确保注塑页里的寸动勾选仍能更新顶部模式文案。

# 模板快速等待界面改版方案
## Summary
`root.panelIndex === 5` 应该对齐 `WaitActionEditor.qml` 的等待逻辑，但用简化布局表达成 3 个互斥模式：
- `单纯延时`
- `X信号`
- `塑机信号`

界面核心是 3 行：
- 一行 `延时文本 + 延时输入框 + 单纯延时复选框`
- 一行 `X信号复选框 + 下拉框`
- 一行 `塑机信号复选框 + 下拉框`

其中这 3 个复选框互斥；`不限时` 不是模式，只是信号等待模式下的附加控制。

## Key Changes
- 保留顶部 `断 / 通 / 上升沿 / 下降沿` 组，但它只在 `X信号` 或 `塑机信号` 模式下显示。
- `延时` 这一行始终显示，结构固定为：
  - 左侧说明文本
  - 中间 `ICConfigEdit`
  - 右侧 `单纯延时` 复选框
- `单纯延时` 选中时：
  - 左侧文本显示 `单纯延时`
  - `X信号`、`塑机信号` 两个复选框都自动取消勾选
  - `X信号`、`塑机信号` 两行的下拉框隐藏
  - `断 / 通 / 上升沿 / 下降沿` 隐藏
  - `不限时` 与 `确认时间` 隐藏
- `X信号` 选中时：
  - `单纯延时`、`塑机信号` 自动取消勾选
  - 延时行左侧文本显示 `延时`
  - `X信号` 下拉框显示并可用
  - `塑机信号` 复选框保留显示，但其下拉框隐藏
  - `断 / 通 / 上升沿 / 下降沿` 显示
  - `不限时` 显示
  - `确认时间` 显示；当 `不限时` 勾选时，`确认时间` 输入框禁用
- `塑机信号` 选中时：
  - 行为与 `X信号` 模式完全一致，只是切换为 `塑机信号` 的下拉框显示并可用
- 信号两行使用明确的 `ICCheckBox + ICComboBox` 组合，不再用 `Rectangle + Text` 模拟控件。
- 3 个模式复选框的互斥逻辑用显式状态联动实现，不复用现有 `waitModeGroup`，避免把 `断 / 通 / 上升沿 / 下降沿` 误纳入同一个模式组。

## Interfaces / State
不新增对外 API；只在面板内部明确这些状态：
- `waitDelayBox.isChecked` 表示 `单纯延时` 模式
- `waitXSignalBox.isChecked` 表示 `X信号` 模式
- `waitMachineSignalBox.isChecked` 表示 `塑机信号` 模式
- 派生布尔值：
  - `isSimpleDelayMode`
  - `isXSignalMode`
  - `isMachineSignalMode`

默认值规则：

- 如果有现有动作数据，按数据回填模式
- 如果没有数据，默认进入 `X信号` 模式以贴近 `WaitActionEditor`
- 顶部状态默认保持现有本页习惯值：`断`
- 隐藏的下拉框保留其当前值，不清空

## Test Plan

## Assumptions

## 3. 不建议直接“物理抽离”的原因

`MOLD_Craft.bits` 里的注塑字段和压铸字段是交错排布的，例如：

- `diseFetchStandPointX/Y`
- `injectionTopXMin/XMax`
- `diseFetchStandPointZ/U/V`
- `injectionTopYMin/YMax`
- `diseTopXMin/XMax`
- `injectionTopZMin/ZMax`

后面的状态位也一样交错：

- `injectionTopModEnd`
- `injectionSideModEnd`
- `diecastingCycleStart`
- `...`
- `injectionSideMod`
- `injectionSideAut`
- `...`
- `manualControlX3/X5`
- `...`
- `manualControlY0/Y1/Y5/Y6/Y15`
- `injectionset`

这意味着：

- 直接把注塑字段从 `MOLD_Craft` 抠出来重排，会改掉 bitfield 布局。
- 一旦布局变化，`ICAddr_Mold_Para37 ~ ICAddr_Mold_Para84` 对应的协议地址、模号存储和旧数据兼容性都可能出问题。

## 4. 推荐抽离方式

推荐顺序：**先逻辑抽离，再决定要不要协议重构**。

### 4.1 推荐方案

在目标分支里新增一个注塑视图结构，不动原协议布局，例如：

```cpp
typedef struct {
    uint32_t topXMin;
    uint32_t topXMax;
    uint32_t topYMin;
    uint32_t topYMax;
    uint32_t topZMin;
    uint32_t topZMax;

    uint32_t sideXMin;
    uint32_t sideXMax;
    uint32_t sideYMin;
    uint32_t sideYMax;
    uint32_t sideZMin;
    uint32_t sideZMax;

    uint32_t sideModeLockTimer;
    uint32_t topModeLockTimer;

    uint32_t topMod;
    uint32_t topAut;
    uint32_t topDoor;
    uint32_t topPre;
    uint32_t topPlate;
    uint32_t topModEnd;

    uint32_t sideMod;
    uint32_t sideAut;
    uint32_t sideDoor;
    uint32_t sidePre;
    uint32_t sidePlate;
    uint32_t sideModEnd;

    uint32_t manualX3;
    uint32_t manualX5;
    uint32_t manualY0;
    uint32_t manualY1;
    uint32_t manualY5;
    uint32_t manualY6;
    uint32_t manualY15;

    uint32_t placementType;

```

再配两组函数：

```cpp
void LoadInjectionCraftConfig(const MOLD_Craft* src, InjectionCraftConfig* dst);
void StoreInjectionCraftConfig(MOLD_Craft* dst, const InjectionCraftConfig* src);
```

好处：

- 不破坏原始协议布局。
- 注塑逻辑可以逐步改成依赖 `InjectionCraftConfig`，阅读和复用都更轻。
- 后面如果还要继续抽离压铸，也能用同样方法做 `DiecastCraftConfig`。

### 4.2 如果目标分支只求“最小可运行”

至少要带走这些内容：

- `hccommparagenericdef.h` 中所有注塑字段
- `immboardType`
- `craftType == 6` 注塑入口判断
- `InjectTask.h/.cpp`
- `IOTask.cpp` 里的注塑 EUIO 处理
- `F_CMD_INJECT_WAIT_MODE_OPEN`
- `ALARM_INJECT_UNDEFINED_PLACEMENT_TYPE`
- `ALARM_UNCHECKED_OPEN_MOLD_END`
- `SetType` 与 `injectionset` 的映射

### 4.3 如果目标分支要求“行为尽量一致”

建议一起同步：

- `AutoAction.cpp` 的 `programCheckSum()`
- `GeneralSafety.cpp` 的注塑板安全逻辑
- `display.cpp` 的板卡显示
- `rs48extentioboard.cpp` 的 EU67 扩展板处理
- `ActionBase.h` / `HCTimer.h` / `ActionBase.cpp` 里的 `immboardType` 镜像

## 5. 本次梳理里明确不算“注塑专用”的内容

这些内容和注塑在同一个大结构里，但不建议作为“注塑专用抽离”目标：

- `MOLD_PRO_USE`
- `MOLD_P`
- `MOLD_UnsafeArea`
- `targetFollow*`
- `encoder*`

- `dise*`
