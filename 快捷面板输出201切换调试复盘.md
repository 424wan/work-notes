# 快捷面板输出 201 切换调试复盘

## 1. 问题起点 / 需求背景

### 最初的问题或需求是什么

这次的目标不是单纯“改个值”，而是要把 Easy Program 里几个快捷插入项的整条链路摸清楚：

- 喷油
- 传送带
- 预留1~4

用户最初的问题是两层：

1. 先确认这些界面项插入后，实际走的是哪个 `action`
2. 再顺着 `hccommparagenericdef.h` 追到主机里，确认最终执行的是哪个指令类和函数

在这个基础上，后面又新增了一个明确目标：

- 让这几个快捷插入项从当前的 `200` 改成走 `201`

### 初始现象或期望效果

界面上这几个项都有两个输入：

- 动作时间
- 间隔次数

从产品直觉上看，很容易让人以为它们应该是“间隔输出”这一类能力，也就是：

- 隔多少次触发一次
- 每次输出持续多久

但在没追代码之前，这只是猜测，不能直接下结论。

### 当时最可能的几个原因或实现思路

最开始我认为有 3 种可能：

1. 这几个快捷项其实已经走 `201`，只是界面文案没有明确写出来  
   这是最理想情况，因为只需要确认，不一定要改代码。

2. 界面显示了“间隔次数”，但底层实际上还是走普通输出 `200`  
   这种情况很常见，属于 UI 已经长出来了，但生成 action 的逻辑没跟上。

3. HMI 侧是 `200`，主机侧再把它转成某种“间隔输出”  
   这种情况也不能排除，因为有些系统会在协议层和执行层做二次转换。

所以一开始不能直接改，必须先把“界面生成什么”和“主机实际执行什么”分开追。

## 2. 分析与排查过程

### 步骤 1：先确认界面上这几个项分别对应哪个点位

- 当时的判断是什么  
  如果连“喷油/传送带/预留1~4”分别对应哪个 Y 点都没搞清楚，后面追主机就会飘。第一步必须先把界面选择项和具体点位绑定关系找出来。

- 为什么这么判断  
  因为主机执行层不会认“喷油”这种中文名，它只认 `type / point / action`。所以必须先把中文项翻译成真正的 IO 点。

- 做了什么验证  
  看了 [EasyProgramFunctionPanel.qml](/home/harry/robot/HMI-RX_/qml/App_HAMOUI/teach/EasyProgramFunctionPanel.qml) 的数据定义。

- 结果如何  
  很快确认：
  - 喷油 -> `Y025`
  - 传送带 -> `Y033`
  - 预留1~4 -> `Y034 ~ Y037`

这一步的价值是把“产品语言”转换成了“代码语言”。

### 步骤 2：确认这些项插入时到底生成哪个 action

- 当时的判断是什么  
  关键不在 UI 长什么样，而在点击“插入”时最终调了哪个生成函数。

- 为什么这么判断  
  同一个界面完全可以显示“动作时间”“间隔次数”，但只要生成函数还是普通输出，那主机看到的就还是普通输出。

- 做了什么验证  
  继续查看 [EasyProgramFunctionPanel.qml](/home/harry/robot/HMI-RX_/qml/App_HAMOUI/teach/EasyProgramFunctionPanel.qml) 里：
  - `createReserveSignalQuickAction()`
  - `createAuxiliaryQuickAction()`
  - `createOutputQuickAction()`

- 结果如何  
  确认这两类快捷项最后都走到了 `createOutputQuickAction()`，而它内部调用的是：

  - `ExtentActionDefine.extentOutputAction.generateActionObject(...)`

  这说明它们生成的是普通输出动作，不是间隔输出动作。

这一步第一次把问题收紧到了“界面生成链路”。

### 步骤 3：确认 `extentOutputAction` 对应的 action 编号

- 当时的判断是什么  
  虽然已经看到了函数名是 `extentOutputAction`，但还要再确认一次它对应的 action 数字，避免名字误导。

- 为什么这么判断  
  在老项目里，名字和实际协议号不一致是常见问题。必须看定义，不要靠经验猜。

- 做了什么验证  
  看了 [ExtentActionDefine.js](/home/harry/robot/HMI-RX_/qml/App_HAMOUI/teach/extents/ExtentActionDefine.js)。

- 结果如何  
  明确看到：
  - `extentOutputAction.action = 200`
  - `extentIntervalOutputAction.action = 201`

到这一步，界面侧的结论已经非常明确：

- 当前喷油 / 传送带 / 预留1~4 插入时，走的是 `200`
- 并没有走 `201`

### 步骤 4：确认“间隔次数”是不是只是显示了，但根本没被用到

- 当时的判断是什么  
  既然生成的是 `200`，那界面上的“间隔次数”很可能只是一个没接线的输入框。

- 为什么这么判断  
  这是一个很高概率的“半成品 UI”特征：界面有控件，但没有参与业务逻辑。

- 做了什么验证  
  查了 [EasyProgramFunctionPanel.qml](/home/harry/robot/HMI-RX_/qml/App_HAMOUI/teach/EasyProgramFunctionPanel.qml) 里这两个输入框附近的代码，重点看：
  - 它们有没有 `id`
  - 创建 action 时有没有读取它们
  - 回填 action 时有没有写回它们

- 结果如何  
  结论很直接：
  - 这两个“间隔次数”输入框原来没有 `id`
  - 创建 action 时没有读取它们
  - 回填时也没有写回它们

所以这个字段在当前实现里是“显示了，但没生效”。

这是一个很关键的中间结论，因为它解释了为什么用户会觉得“界面像是支持间隔输出”，但系统行为却不像。

### 步骤 5：把 HMI 结论和主机结论对上，避免只看半边

- 当时的判断是什么  
  还不能只停留在“界面生成了 200”，必须继续确认主机收到 `200` 后到底执行什么，防止主机内部再做特殊分发。

- 为什么这么判断  
  这一步是为了排除第三种可能：HMI 虽然发 `200`，但主机内部把它特殊处理成了“间隔输出”。

- 做了什么验证  
  顺着主机代码看：
  - [hccommparagenericdef.h](/home/harry/robot/EC-APRobotHost_/ec_develop/Include/hmi/protocol/hccommparagenericdef.h)
  - [data.cpp](/home/harry/robot/EC-APRobotHost_/ec_develop/Src/module/data.cpp)
  - [IOOutput.h](/home/harry/robot/EC-APRobotHost_/ec_develop/instruction/IOOutput.h)
  - [IOOutput.cpp](/home/harry/robot/EC-APRobotHost_/ec_develop/instruction/IOOutput.cpp)
  - [IOIntervalOutput.h](/home/harry/robot/EC-APRobotHost_/ec_develop/instruction/IOIntervalOutput.h)
  - [IOIntervalOutput.cpp](/home/harry/robot/EC-APRobotHost_/ec_develop/instruction/IOIntervalOutput.cpp)

- 结果如何  
  主机侧结论也非常清楚：
  - `F_CMD_IO_OUTPUT = 200` -> `new IOOutput`
  - `F_CMD_IO_INTERVAL_OUTPUT = 201` -> `new IOIntervalOutput`

并且当前这几个快捷项因为发的是 `200`，所以最终执行链是：

- `IOOutput::process()`
- `IOOperation()`
- `SetIO()`

这一步把“界面推测”变成了“端到端确认”。

### 步骤 6：解释“动作时间”和“间隔次数”为什么会让人误解

- 当时的判断是什么  
  光知道是 `200` 还不够，还要解释清楚“动作时间”在当前链路里到底是什么意思，不然用户会以为只是 action 号不对。

- 为什么这么判断  
  真正容易误导人的不是 `200/201` 这个数字，而是界面文案和实际执行语义不一致。

- 做了什么验证  
  继续看 `IOOutput` 和 `IOIntervalOutput` 的处理逻辑，尤其是：
  - `delay`
  - `cnt`
  - 时间输出分支

- 结果如何  
  我确认了两件事：

  1. 当前这几个快捷项走 `200` 时，`动作时间` 实际更接近“延时后输出”，不是“保持导通多久”
  2. `间隔次数` 根本没有被带入 action，所以主机完全不会用它

这一步的作用是把“为什么用户感觉不对”讲清楚了。

### 步骤 7：判断主机要不要改，还是 HMI 改就够

- 当时的判断是什么  
  在确认现状之后，下一步不是立刻写代码，而是判断“修复点应该放在哪一层”。

- 为什么这么判断  
  如果主机本来就已经支持 `201`，那再去改主机不但没必要，还会扩大改动面。

- 做了什么验证  
  查 HMI 里其他地方有没有已经在用 `201`，以及主机是否完整支持。

- 结果如何  
  我找到了现成参考：
  - [InjectTechnology.qml](/home/harry/robot/HMI-RX_/qml/App_HAMOUI/teach/InjectTechnology.qml)

  那里已经在正常生成 `extentIntervalOutputAction(201)`。这说明：

  - HMI 里已有现成的 201 用法
  - 主机侧的 201 也已经是通的

因此可以得出一个很重要的工程判断：

- 这不是“协议能力缺失”
- 这是“快捷面板没有接到已有能力上”

### 步骤 8：先设计“最小修改”，再设计“最小闭环”

- 当时的判断是什么  
  用户问“要改什么才能让它走 201”，这里不能直接给出最大方案，应该先给最小方案，再判断是否够用。

- 为什么这么判断  
  工程上“能发出去”和“用户体验完整”是两回事。

- 做了什么验证  
  我把改动拆成两层：

  第一层，只让新插入动作走 `201`：
  - 给“间隔次数”输入框加 `id`
  - 创建函数改成生成 `201`

  第二层，保证这条链是完整可编辑的：
  - 识别已有 `201`
  - 回填 `cnt`

- 结果如何  
  最后确定采用“最小闭环”而不是“最小能跑”：

  因为如果只改创建，不改回显，用户插入后再点编辑会发现这条指令识别不回来，这属于明显的半修复。

### 步骤 9：实际修改时只动快捷面板，不碰主机

- 当时的判断是什么  
  既然问题根因已经定位在 HMI 快捷面板，那代码修改就应该集中在那里，避免连带修改主机。

- 为什么这么判断  
  改动范围越小，风险越低；而且主机在这件事上不是问题点。

- 做了什么验证  
  在 [EasyProgramFunctionPanel.qml](/home/harry/robot/HMI-RX_/qml/App_HAMOUI/teach/EasyProgramFunctionPanel.qml) 中做了 5 个点的改动：

  1. 新增 `createIntervalOutputQuickAction()`
  2. `createReserveSignalQuickAction()` 改走 `201`
  3. `createAuxiliaryQuickAction()` 改走 `201`
  4. `resolveQuickPanelIndex()` 识别 `201`
  5. 预留 / 辅助设备回填时补上 `cnt`

  同时给两个“间隔次数”输入框补了 `id`。

- 结果如何  
  代码层面已经形成完整闭环：
  - 新插入会生成 `201`
  - 重新加载已有动作时也能识别
  - 界面可回填 `cnt`

## 3. 过程中出现的新问题

这次没有遇到“改完直接报错”的运行时问题，但在分析和修改过程中，暴露出了两个非常典型的次生问题。

### 新问题 1：界面有“间隔次数”，但业务逻辑没接上

- 出现原因是什么  
  UI 控件已经存在，但没有参与 action 生成，也没有参与回填。

- 本质属于什么问题  
  `UI问题 + 逻辑问题`

- 为什么重要  
  这种问题最容易误导排查方向。用户会天然认为“界面上有这个字段，就应该已经生效了”，但代码并不一定这么实现。

### 新问题 2：如果只改创建为 201，编辑已有指令会失效

- 出现原因是什么  
  `resolveQuickPanelIndex()` 原来只认 `200`，并且回填函数也只处理 `delay`，不处理 `cnt`。

- 本质属于什么问题  
  `状态问题 + UI问题`

- 为什么重要  
  这是很多“最小改动”常见的坑：创建链改了，但识别链没改，导致系统进入一种“能插入、不能回填”的半残状态。

### 新问题 3：动作时间的语义和界面文案不一致

- 出现原因是什么  
  当前这几个快捷项原来走 `200`，而 `200` 在这条链路里更像“延时后输出”，不是“保持输出时间”。

- 本质属于什么问题  
  `逻辑问题`

- 为什么重要  
  如果不把这个语义差异说清楚，后面改成 `201` 时用户会以为只是“换了 action 号”，但实际上连时间参数的含义都更接近原本期望了。

## 4. 关键转折点

### 哪一步让问题真正接近解决

真正的关键转折点有两个，但最重要的是第二个：

1. 确认这几个快捷项实际生成的是 `200`
2. 确认主机早就已经支持 `201`，并且 HMI 里别的页面已经在正常使用 `201`

### 为什么这一点是关键

第一步解决的是“现状不清楚”的问题。  
第二步解决的是“修复点到底在哪”的问题。

一旦确认：

- 主机支持没问题
- HMI 里也有现成写法

那么整个问题就从“大系统联调问题”收缩成了：

- 快捷面板没有接到已有能力上

这会让后面的改动决策非常干净，不会误入主机侧大改。

### 如果没有这一步，会一直卡在哪

如果没有把“主机已支持 201”这件事确认下来，后面很可能会卡在两个错误方向：

1. 怀疑主机协议不支持，需要补协议
2. 怀疑 `201` 会牵扯一堆底层逻辑，不敢动

这两条都是无效放大。

## 5. 最终解决方案

### 最终是怎么修复的

最终修复思路很简单：

- 不改主机
- 只改快捷面板
- 让“预留1~4 / 喷油 / 传送带”从原来的普通输出 `200` 切到间隔输出 `201`
- 同时补齐已有 `201` 指令的识别和回填

### 涉及哪几个关键点

1. 创建 action 时改走 `extentIntervalOutputAction(201)`，并把 `cnt` 传进去
2. 让快捷面板在识别已有动作时也接受 `201`
3. 回填时把 `cnt` 写回“间隔次数”输入框

## 6. 如果重新来一遍，最快路径是什么

### 可以跳过哪些无效尝试

如果重新来一遍，其实可以少走一点弯路：

- 不用一开始就去怀疑主机协议缺失
- 不用先想“是不是要新增一个新 action”
- 不用先改主机

因为 HMI 里已经存在 `extentIntervalOutputAction(201)`，这是一个很强的信号，说明优先级应该是“先看复用，不是先看发明”。

### 最优排查顺序应该是怎样

最优顺序应该是：

1. 先看快捷面板生成 action 的入口函数
2. 确认它到底生成 `200` 还是 `201`
3. 查 HMI 里有没有现成页面已经在用 `201`
4. 再确认主机 `201` 是否完整支持
5. 最后只改“创建 + 识别 + 回填”这三块

这个顺序的优点是：

- 每一步都在缩小范围
- 不会一开始就陷入底层细节
- 很容易做出“最小闭环修改”

## 7. 本质总结

### 这个问题的本质原因一句话总结

本质原因就是：

- 快捷面板的 UI 已经长成了“间隔输出”的样子，但生成和回填逻辑还停留在“普通输出 `200`”上。

### 以后遇到类似问题，可以优先检查什么

以后遇到类似问题，优先检查这 3 件事：

1. UI 上显示出来的字段，是否真的参与了 action 生成  
   这是最容易出现“看起来支持，实际上没接线”的地方。

2. 创建链和回填链是不是同时改了  
   很多问题不是“不能创建”，而是“创建了以后不能再编辑”。

3. 项目里有没有别的页面已经实现了同类能力  
   老项目里最快的路径通常不是自己重新设计，而是先找已有实现再复用。

如果再用一句更偏工程化的话总结这次过程，就是：

- 不要先改代码，先确认“问题到底发生在生成层、协议层，还是执行层”。
