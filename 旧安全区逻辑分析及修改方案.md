# 注塑安全区现状梳理与 AOB+H 改造建议

> 更新说明：当前工作区已经在 `InjectTask.cpp` 落了第一版改动，开始直接读取 `easy_program_safe_o/a/b/h` 并转换成运行时使用的旧版 `x/y/z min/max` 边界。下文第 2、3 节主要保留改造前的现状分析，方便回溯原逻辑。

## 1. 结论先看

当前运行中的注塑安全区逻辑，仍然是“按安装方式选择一组 `x/y/z min/max` 包围盒参数，再按当前点位做区域判定”。

代码里已经接入了 `easy_program_install_type` 作为安装方式来源，但没有使用协议中新增的 `easy_program_safe_o/a/b/h` 字段做安全区计算。也就是说：

1. 安装方式已经走“简单编程”的新字段。
2. 安全区几何本体仍然走旧的 `injectionTop* / injectionSide*` 六个边界值。
3. `easy_program_safe_o/a/b/h`、`easy_program_safe_inner_point_*`、`easy_program_safe_outer_point_*` 目前只在协议定义里看到，未发现参与运行时判定。

### 2.4 区域状态含义

状态枚举定义在 [`ec_develop/action/InjectTask.h`](/home/harry/robot/EC-APRobotHost_/ec_develop/action/InjectTask.h#L33)：

1. `SAFE_AREA_SAFE_INMODE`：模内
2. `SAFE_AREA_SAFE_READY`：取料待机区/准备点
3. `SAFE_AREA_SAFE_TRANS`：过渡区
4. `SAFE_AREA_SAFE_X_OUT`
5. `SAFE_AREA_SAFE_Y_OUT`
6. `SAFE_AREA_SAFE_Z_OUT`
7. `SAFE_AREA_OTHERAREAS`：其它不安全区域

这几个状态不是单纯“是否在包围盒内”，而是把包围盒外不同方向分开了，后面报警逻辑就依赖这个分类。

### 2.5 四种安装方式分别怎么判

判定代码在 [`ec_develop/action/InjectTask.cpp`](/home/harry/robot/EC-APRobotHost_/ec_develop/action/InjectTask.cpp#L416)。

#### 顶装 `TYPE_TOP`

可理解为：

1. `z > z_Max` 且 `x/y` 在范围内：`SAFE_AREA_SAFE_READY`
2. `z > z_Max` 但 `x/y` 不在范围内：`SAFE_AREA_SAFE_TRANS`
3. `z_Min <= z <= z_Max` 且 `x/y` 在范围内：`SAFE_AREA_SAFE_INMODE`
4. `z < z_Min` 且 `x/y` 在范围内：`SAFE_AREA_SAFE_Z_OUT`
5. `z < z_Min` 且 `x/y` 不在范围内：`SAFE_AREA_OTHERAREAS`

也就是说，顶装默认把 `+Z` 一侧当作“模外待机方向”，把 `-Z` 更深处当作越界。

#### 侧装 `TYPE_SIDE_X_PP`

可理解为：

1. `x < x_Min` 且 `y/z` 在范围内：`SAFE_AREA_SAFE_READY`
2. `x < x_Min` 但 `y/z` 不在范围内：`SAFE_AREA_SAFE_TRANS`
3. `x_Min <= x <= x_Max` 且 `y/z` 在范围内：`SAFE_AREA_SAFE_INMODE`
4. `x > x_Max` 且 `y/z` 在范围内：`SAFE_AREA_SAFE_X_OUT`
5. `x > x_Max` 且 `y/z` 不在范围内：`SAFE_AREA_OTHERAREAS`

也就是把 `-X` 一侧当模外准备区，`+X` 深处当模内更深方向。

#### 侧装 `TYPE_SIDE_Y_PP`

可理解为：

1. `y < y_Min` 且 `x/z` 在范围内：`SAFE_AREA_SAFE_READY`
2. `y < y_Min` 但 `x/z` 不在范围内：`SAFE_AREA_SAFE_TRANS`
3. `y_Min <= y <= y_Max` 且 `x/z` 在范围内：`SAFE_AREA_SAFE_INMODE`
4. `y > y_Max` 且 `x/z` 在范围内：`SAFE_AREA_SAFE_Y_OUT`
5. `y > y_Max` 且 `x/z` 不在范围内：`SAFE_AREA_OTHERAREAS`

#### 侧装 `TYPE_SIDE_Y_RP`

和 `TYPE_SIDE_Y_PP` 相反：

1. `y > y_Max` 且 `x/z` 在范围内：`SAFE_AREA_SAFE_READY`
2. `y > y_Max` 但 `x/z` 不在范围内：`SAFE_AREA_SAFE_TRANS`
3. `y_Min <= y <= y_Max` 且 `x/z` 在范围内：`SAFE_AREA_SAFE_INMODE`
4. `y < y_Min` 且 `x/z` 在范围内：`SAFE_AREA_SAFE_Y_OUT`
5. `y < y_Min` 且 `x/z` 不在范围内：`SAFE_AREA_OTHERAREAS`

### 2.6 报警和 IO 怎么联动

报警状态机会在 `SafeAreaAlarmInMode()` 里处理，见 [`ec_develop/action/InjectTask.cpp`](/home/harry/robot/EC-APRobotHost_/ec_develop/action/InjectTask.cpp#L323)。

核心行为：

1. `lastSafeArea` 记录“上一次区域”，用于判断从哪里走向哪里。
2. 手动模式下，只有“已设置安全区”时才报相关安全区报警；自动模式全程参与。
3. 一旦形成报警，会 `EmergencyStop()`。
4. `GetIO(0,14)` 模内检测信号断开时，直接把当前位置视作 `SAFE_AREA_SAFE_INMODE`。
5. 当判定为 `SAFE_AREA_SAFE_INMODE` 时，会把 `SetIO(5,17,1)`，否则清掉。

这说明旧逻辑不只是给 UI 看区域，而是直接参与：

1. 报警
2. 急停
3. 可关模等信号输出约束
4. 模内状态保持

### 2.7 为什么可以从安全区内部退回 `READY`，但不能更深进入或从侧面跑出

这个行为不是由运动规划器“把路径夹在区域里”实现的，而是由下面两层逻辑共同实现：

1. `compareSafeAreaValue()` 先把当前位置分成 `READY / INMODE / X_OUT / Y_OUT / Z_OUT / TRANS / OTHERAREAS`
2. `SafeAreaAlarmInMode()` 再根据 `lastSafeArea -> ret` 的状态迁移关系决定是否报警并急停

#### 第一步：先把“退回前表面外侧”和“向更深处走”分成不同状态

以顶装为例，判定代码见 [`ec_develop/action/InjectTask.cpp`](/home/harry/robot/EC-APRobotHost_/ec_develop/action/InjectTask.cpp#L499)：

1. `z > z_Max` 且 `x/y` 还在矩形投影内，会被判成 `SAFE_AREA_SAFE_READY`
2. `z_Min <= z <= z_Max` 且 `x/y` 在范围内，会被判成 `SAFE_AREA_SAFE_INMODE`
3. `z < z_Min` 且 `x/y` 在范围内，会被判成 `SAFE_AREA_SAFE_Z_OUT`
4. 深度还在范围内、但 `x/y` 从侧边跑出去时，会被判成 `SAFE_AREA_SAFE_X_OUT` 或 `SAFE_AREA_SAFE_Y_OUT`

所以对顶装来说：

1. 朝 `AOB` 面外侧退出来，是 `READY`
2. 朝更深的模内方向继续进去，是 `Z_OUT`
3. 从左右侧边跑出投影范围，是 `X_OUT / Y_OUT`

侧装三种类型也是同一个思路，只是“前后方向”对应的轴不同，代码在 [`ec_develop/action/InjectTask.cpp`](/home/harry/robot/EC-APRobotHost_/ec_develop/action/InjectTask.cpp#L522) 到 [`ec_develop/action/InjectTask.cpp`](/home/harry/robot/EC-APRobotHost_/ec_develop/action/InjectTask.cpp#L580)：

1. 朝前表面外侧退出，会被判成 `READY`
2. 朝模内更深方向进去，会被判成对应的深度越界状态
3. 从表面矩形的侧边跑出去，会被判成另外两个轴向的越界状态

这一步的本质是：代码先把“原路退回”和“非法越界”分成了不同的 `ret`。

#### 第二步：状态机只放行 `INMODE -> READY`，不放行 `INMODE -> *_OUT`

真正决定“是否允许”是在 `SafeAreaAlarmInMode()`，见 [`ec_develop/action/InjectTask.cpp`](/home/harry/robot/EC-APRobotHost_/ec_develop/action/InjectTask.cpp#L406)。

当 `lastSafeArea == SAFE_AREA_SAFE_INMODE` 时，代码只对下面几种结果报错：

1. `SAFE_AREA_SAFE_TRANS`
2. `SAFE_AREA_OTHERAREAS`
3. `SAFE_AREA_SAFE_X_OUT`
4. `SAFE_AREA_SAFE_Y_OUT`
5. `SAFE_AREA_SAFE_Z_OUT`

这里有一个很关键的点：`SAFE_AREA_SAFE_READY` 不在报警分支里。

所以从 `INMODE` 退回到 `READY` 时：

1. `compareSafeAreaValue()` 会把点判成 `READY`
2. `SafeAreaAlarmInMode()` 不会因为 `INMODE -> READY` 这次迁移报警
3. 如果当前点不是目标点并且没报警，`lastSafeArea` 会更新为新的 `READY`，见 [`ec_develop/action/InjectTask.cpp`](/home/harry/robot/EC-APRobotHost_/ec_develop/action/InjectTask.cpp#L476)

反过来，从 `INMODE` 继续朝更深方向走，或者从侧面跑出时：

1. `compareSafeAreaValue()` 会返回 `X_OUT / Y_OUT / Z_OUT`
2. `SafeAreaAlarmInMode()` 会把它们映射成 `ALARM_BEYOND_*_SAFETY_ZONE`
3. 满足安全区启用条件时会立即 `EmergencyStop()`，见 [`ec_develop/action/InjectTask.cpp`](/home/harry/robot/EC-APRobotHost_/ec_develop/action/InjectTask.cpp#L470)

#### 第三步：因此形成了一条“合法通道”

把上面两层合起来看，当前逻辑等价于一条状态通道：

1. `READY -> INMODE`：允许进入安全区内部
2. `INMODE -> READY`：允许原路退回前表面外侧
3. `INMODE -> X_OUT / Y_OUT / Z_OUT / OTHERAREAS`：不允许，会报警或急停

所以更准确的说法不是“进入后会被硬限制在这个区域中”，而是：

1. 代码允许机器人在 `READY` 和 `INMODE` 之间往返
2. 代码不允许机器人从 `INMODE` 再向更深处进入，或从矩形侧边跑出
3. 这种限制是靠运行时区域判定 + 报警急停实现的，不是轨迹规划层面的硬夹持

#### 一个容易忽略的细节

顶装分支在“深度还在范围内，但 `x/y` 越界”时，只有 `lastSafeArea == SAFE_AREA_SAFE_INMODE` 才会明确判成 `X_OUT / Y_OUT`，见 [`ec_develop/action/InjectTask.cpp`](/home/harry/robot/EC-APRobotHost_/ec_develop/action/InjectTask.cpp#L517)。

这也说明当前实现本身就是带状态记忆的，不是单纯的几何点内外判断；它依赖 `lastSafeArea` 来区分“正在从模内退出”还是“本来就在外侧其它区域”。

### 3.1 新协议字段还没接进运行逻辑

7. `easy_program_safe_outer_point_x/y/z`

但在仓库内全文搜索后，除了 `easy_program_install_type` 被 `InjectTask.cpp` 使用之外，其余字段未发现运行时引用。

### 3.2 旧边界字段定义位置暂未找到

`InjectTask.cpp` 里明确使用了：

1. `injectionTopXMin / Max`
2. `injectionTopYMin / Max`
3. `injectionTopZMin / Max`
4. `injectionSideXMin / Max`
5. `injectionSideYMin / Max`
6. `injectionSideZMin / Max`

但当前仓库全文搜索，暂时没有搜到这些字段的定义位置，只搜到了使用点。这件事在正式改造前需要确认：

1. 是协议头被替换了，但旧成员仍由别名或生成代码提供？
2. 还是这段代码本身已经落后于当前协议定义？

如果这里不先确认，后面直接改计算逻辑容易改偏。

### 3.3 顶装分支疑似存在一处轴变量笔误

在顶装判定里，第二个越界分支写的是：

```cpp
else if((p[0] < sfConfig.s.y_Min || p[0] > sfConfig.s.y_Max) && (lastSafeArea == SAFE_AREA_SAFE_INMODE))
```

这里看起来应当是 `p[1]` 对 `y_Min / y_Max` 做比较，而不是 `p[0]`。位置见 [`ec_develop/action/InjectTask.cpp`](/home/harry/robot/EC-APRobotHost_/ec_develop/action/InjectTask.cpp#L443)。

这不是这次 AOB+H 改造的主目标，但如果后续还复用旧状态机，这个点建议一起确认。

### 3.4 顶装在部分“平面内越界”场景下可能落不到明确状态

顶装分支在 `z_Min <= z <= z_Max`、但 `x/y` 不在平面范围内且 `lastSafeArea != SAFE_AREA_SAFE_INMODE` 时，`ret` 有机会保持初值 `SAFE_AREA_SAFE`。这和侧装分支“必然给出某类状态”的写法不完全一致。

这个行为可能是历史兼容，也可能是遗漏，建议在新方案设计时顺手统一。

## 4. 新方案怎么实施更稳

## 4.1 推荐原则

建议把这次改造拆成“两层”：

1. 输入层：从 `O/A/B/H` 读取并缓存新的安全区定义
2. 判定层：把运行时点位映射到统一的“面内坐标 + 深度坐标”后，再复用现有状态机语义

这样做的好处是：

1. UI/协议改动和运行逻辑解耦
2. 后续如果 `AOB` 真要支持非纯轴对齐，也还有扩展空间
3. 可以先做兼容版，降低一次性重写风险

## 4.2 最小风险方案：先把 AOB+H 转成旧包围盒

如果你们这次的新几何规则本质上仍然是“轴对齐长方体”，只是录入方式改成 `AOB` 面加深度，那么最稳的实现方案是：

1. 新增一个读取 `easy_program_safe_o/a/b/h` 的配置函数
2. 根据 `easy_program_install_type` 把 `AOB+H` 转成 `x/y/z min/max`
3. 旧的 `compareSafeAreaValue()`、`SafeAreaAlarmInMode()` 尽量不动

这样改动面最小，最适合先把功能切过去。

### 顶装

约定 `AOB` 是上表面，`H` 朝模内方向：

1. `x/y` 范围由 `A/O/B` 所在顶面矩形决定
2. `z_Max` 取 `AOB` 面高度
3. `z_Min = z_Max - H`

### 侧装 X+

约定 `AOB` 是前表面，`H` 朝模内方向：

1. `y/z` 范围由前表面矩形决定
2. `x_Min` 取前表面位置
3. `x_Max = x_Min + H`

### 侧装 Y+

1. `x/z` 范围由前表面矩形决定
2. `y_Min` 取前表面位置
3. `y_Max = y_Min + H`

### 侧装 Y-

1. `x/z` 范围由前表面矩形决定
2. `y_Max` 取前表面位置
3. `y_Min = y_Max - H`

这个方案的关键不是“把 8 个顶点全算出来”，而是先明确哪一个面是“模外那一面”，以及 `H` 的正方向到底是不是固定指向模内。

## 4.3 更长期的方案：直接做 AOB 面 + 深度判定

如果后续希望不再受 `x/y/z min/max` 束缚，可以直接改成棱柱判定：

1. `u = A - O`
2. `v = B - O`
3. `n = inward_normal`
4. 对任意点 `P` 计算 `P - O` 在 `u/v/n` 三个方向上的投影
5. `u/v` 投影用于判断是否落在 `AOB` 面内
6. `n` 投影用于判断是在准备区、模内还是深度越界

但这个版本要先把协议语义定清楚，否则实现时会有歧义。

## 4.4 新协议在实现前必须先定死的规则

这几项建议先和 HMI/工艺侧统一，不然代码里会不断猜：

1. `O` 是否一定是矩形面的公共角点？
2. `A`、`B` 是否一定分别是从 `O` 出发的两条邻边端点？
3. `OA` 与 `OB` 是否保证正交？
4. `H` 是否总是正数？
5. `H` 的方向是由安装方式决定，还是由 `O/A/B` 点序右手法则决定？
6. 侧装三种类型虽然都说“面朝前”，但深度正方向是否分别对应 `+X / +Y / -Y`？
7. `easy_program_safe_inner_point_*` 和 `easy_program_safe_outer_point_*` 后续是否要参与校验或只是教导辅助点？

我个人建议第一版把规则定成：

1. `O` 是矩形公共角点
2. `A`、`B` 是与 `O` 相邻的两个角点
3. `H` 只允许正数
4. 深度方向完全由 `easy_program_install_type` 决定

这样代码最直观，也最不容易在现场教导时出现“同样一组点，换个录入顺序结果反了”的问题。

## 5. 建议的改造步骤

### 第一步：把数据源切对

新增一个新的安全区配置结构，例如：

```cpp
typedef struct {
    FLOAT32 o[3];
    FLOAT32 a[3];
    FLOAT32 b[3];
    FLOAT32 h;
} SafeAreaPlaneConfig;
```

然后提供：

1. `isSafeAreaSettingByPlane()`
2. `getSafeAreaPlaneConfig()`

都从 `easy_program_safe_*` 读取。

### 第二步：先做“新输入 -> 旧包围盒”的转换

新增一个转换函数，例如：

```cpp
BOOL buildLegacySafeAreaFromPlane(UINT16 type, const SafeAreaPlaneConfig* plane, SafeAreaConfig* box);
```

把 `AOB+H` 先映射成旧 `x/y/z min/max`，这样可以快速验证：

1. 旧报警是否保持一致
2. 自动/手动行为是否保持一致
3. `lastSafeArea` 状态机是否还能复用

### 第三步：补校验和防呆

建议在读取新配置时加以下校验：

1. `OA` 和 `OB` 不能是零向量
2. `H > 0`
3. 顶装时 `AOB` 应接近水平面
4. 侧装时 `AOB` 应接近前立面
5. 如果点位明显不满足当前安装方式约束，直接报配置错误，不进入运行时安全判定

### 第四步：回归验证

建议至少做下面几组验证：

1. 顶装：面上方应判 `READY`
2. 顶装：面内且深度内应判 `INMODE`
3. 顶装：超过深度应判 `Z_OUT`
4. 侧装 X+：前方应判 `READY`
5. 侧装 X+：更深处应判 `X_OUT`
6. 侧装 Y+：前方应判 `READY`
7. 侧装 Y-：前方应判 `READY`
8. 手动未设置安全区时，不应误报
9. 自动模式切换时，`needReflashArea` 和 `lastSafeArea` 行为要与现在一致

## 6. 我建议的实际落地顺序

如果下一步就要开始改代码，我建议按这个顺序做：

1. 先确认 `O/A/B/H` 的几何语义和 `H` 方向规则
2. 再确认旧 `injectionTop/Side*Min/Max` 字段目前是否还真有数据来源
3. 第一版只做“新字段读取 + 转旧包围盒”
4. 跑通后再考虑是否要把 `compareSafeAreaValue()` 升级成真正的 AOB 棱柱判定

这个顺序的好处是，既能尽快切到新录入方式，也不会一次把报警状态机全重写掉。
