# InjectTechnology 中 `checkSignal` 与“正反向检测设置”的代码联合分析

## 1. 问题背景

在注塑工艺页面 `InjectTechnology.qml` 中，用户会看到两组看起来很接近的配置：

1. `checkSignal`
   - 例如 `F-1 on check`、`S-1 on check`、`F-1 off check`
2. `directionSelect`
   - 例如 `F-1 on check: reverse/forward`

表面上看，前者已经有了 “on/off check”，似乎已经能表达“夹吸具通/断”的检测；后者再提供一组 `reverse/forward`，容易让人觉得重复。

- 主机代码 `/home/harry/robot/EC-APRobotHost/ec_develop/action/ActionBase.h`
- 主机代码 `/home/harry/robot/EC-APRobotHost/ec_develop/action/InjectTask.cpp`

## 2. 先说结论

`checkSignal` 和 `directionSelect` 不是重复配置，而是同一套“夹/吸确认限检测”功能的两个层次：

- `checkSignal`
  - 决定“要不要对某一路确认限进行检测”
  - 决定“这是开始检测还是结束检测”
  - 决定“延时多久后开始/结束这个检测动作”
- `directionSelect`
  - 决定“该确认限什么电平算成立”
  - 即：输入为 `1` 算有效，还是输入为 `0` 算有效
  - 这就是主机侧的 `xdir`

换句话说：

- `on/off check` 说的是“检测的业务场景”
- `reverse/forward` 说的是“检测成立时的输入极性”

它们不是一个维度。

## 3. `checkSignal` 真正在做什么

### 3.1 生成的是“检测动作”，不是“输出动作”

在 `InjectTechnology.qml` 中，`checkSignal` 被选中后，最终生成的是 `generateFSCheckAction`：

```qml
if(checkSignal.visible){
    for(var i = 0 ;i< checkXModel.count ;i++){
        if(checkXModel.get(i).isSel == 1){
            checkXModel.setProperty(i,"type", statusSelect.checkedIndex)
            checkXModel.setProperty(i,"time", statusDelay.configValue)
            points.push(
                Teach.generateCustomAction(
                    ExtentActionDefine.generateFSCheckAction.generateActionObject(
                        checkXModel.get(i).point,
                        (checkXModel.get(i).type == 0 ? Teach.VALVE_CHECK_START : Teach.VALVE_CHECK_END) - Teach.VALVE_CHECK_END,
                        checkXModel.get(i).time,
                        checkXModel.get(i).status,
                        1
                    )
                )
            );
        }
    }
}
```

对应位置：

- `/home/harry/robot/HMI-RX/qml/App_HAMOUI/teach/InjectTechnology.qml:69-75`

这里很关键的一点是：

- 它生成的是 `FSCheckAction`
- 不是 `extentOutputAction`

所以 `checkSignal` 本质上不是去“控制夹具阀输出通断”，而是去“插入一个确认限检测动作”。

### 3.2 `begin/end` 的真正含义

`checkSignal` 上面有一组：

- `begin`
- `end`

代码里通过 `statusSelect.checkedIndex` 写入 `checkXModel.type`：

- `begin` -> 开始检测
- `end` -> 结束检测

对应：

- `/home/harry/robot/HMI-RX/qml/App_HAMOUI/teach/InjectTechnology.qml:3661-3675`

也就是说，这里的“begin/end”不是夹爪开关本身，而是：

- 从某个时刻开始监控该确认限
- 或在某个时刻停止监控该确认限

### 3.3 `F-1 on check` / `F-1 off check` 的真正含义

在 `checkSignal` 区域中，有两类项目：

- `F-1 on check`、`F-2 on check`、`S-1 on check` ...
- `F-1 off check`、`F-2 off check` ...

对应位置：

- `/home/harry/robot/HMI-RX/qml/App_HAMOUI/teach/InjectTechnology.qml:3695-3856`
- `/home/harry/robot/HMI-RX/qml/App_HAMOUI/teach/InjectTechnology.qml:3861-3931`

这几个名字容易让人误解为：

- on = 输出通
- off = 输出断

但从整体流程看，它们其实表达的是：

- “夹/吸具处于通的业务场景时，对应确认限要怎么检查”
- “夹/吸具处于断的业务场景时，对应确认限要怎么检查”

更准确地说：

- 这些项目属于“确认限检测策略”
- 不是“阀输出控制策略”

这说明系统设计明确区分了两件事：

- 输出动作：控制硬件
- 检测动作：监控确认限输入是否符合预期

## 8. 主机运行时如何判断“检测成立”

注塑任务中，真正执行吸夹确认限检查的是 `GetFSSignalCheck()`：

```cpp

    if(fsSignalFlag[j].xdir == 1) {
        if(GetIO(0, fsSignalFlag[j].id) == false) {
            errPLC = ALARM_F1ON_WITH_F1CONFIRM_OFF + j * 2;
            break;
        }
    }
    else if(fsSignalFlag[j].xdir == 0) {
        if(GetIO(0, fsSignalFlag[j].id)) {
            errPLC = ALARM_F1OFF_WITH_F1CONFIRM_ON + j * 2;
            break;
        }
    }
}
```

对应位置：

- `/home/harry/robot/EC-APRobotHost/ec_develop/action/InjectTask.cpp:544-560`

这段逻辑语义非常明确：

- `xdir == 1`
  - 期望输入为 `1`
  - 如果读到 `0`，就报错
- `xdir == 0`
  - 期望输入为 `0`
  - 如果读到 `1`，就报错

因此：

- `reverse/forward` 最终就是“输入有效电平”的选择
- 不是“输出阀通断”的选择

## 12. 为什么必须保留“正反向检测设置”

### 12.1 因为现场接线不一定同极性

确认限输入在现场可能有两种常见情况：

1. 到位时输入为 `1`
2. 到位时输入为 `0`

这和传感器类型、接线方式、板卡定义有关。

如果系统只提供：

- “F-1 on check”

而没有：

- “这一项是高电平有效还是低电平有效”

那么系统无法知道：

- 夹具动作后，到底应该期待输入变成 `1` 还是 `0`

### 12.2 不做极性配置会造成大量误报

如果现场确认限是低电平有效，但程序按高电平有效去检查，就会出现：

- 夹具明明已经夹住
- 确认限其实已经到位
- 但程序仍然报警

这正是“正反向检测设置”存在的意义。

它不是“再做一次通断控制”，而是“定义这个检测动作的判定标准”。

## 14. 一个容易混淆的点

`F-1 on check` 这个名字本身非常容易误导成：

- “F-1 输出通”

但从代码和主机协议看，它更准确的含义其实应理解为：

- “在 F-1 处于 on 场景时，对 F-1 确认限进行检测”

同理：

- `F-1 off check`
  - 不是“把 F-1 关掉”
  - 而是“在 F-1 处于 off 场景时，对 F-1 确认限进行检测”

也就是说：

- `on/off`
  - 代表业务场景
- `reverse/forward`
  - 代表输入有效极性

## 16. 最终结论

### 16.1 回答原问题

“`checkSignal` 不是已经可以设置夹吸具通或者断了吗，后面那里又弄个正反向检测设置干嘛？”

准确回答是：

- `checkSignal` 不是在设置夹吸具输出通断
- 它是在设置“是否对某一路确认限启用检测，以及在什么阶段启用/关闭”
- 后面的“正反向检测设置”是在设置“这一路确认限什么电平算检测成立”

### 16.2 两者的职责分工

- 输出通断
  - 由 `extentOutputAction` 负责
- 检测启停
  - 由 `checkSignal` 负责
- 检测极性
  - 由 `directionSelect` 负责

### 16.3 为什么必须这样设计

因为“阀输出状态”和“确认限输入状态”不是同一个信号：

- 一个是控制量
- 一个是反馈量

而反馈量在现场又可能是高有效或低有效，所以必须保留“正反向检测设置”。

## 17. 建议

  - 例如“低电平有效 / 高电平有效”
  - 或“断有效 / 通有效”
- 把 `F-1 on check` 这类名字改成更明确的描述
  - 例如“F-1 通状态确认检测”
  - “F-1 断状态确认检测”

这样会比现在的命名更贴近主机真实语义。
